# /finegrained-check

对目标设计文档做细粒度一致性检查。用法：`/finegrained-check <文档路径>`

## 执行步骤

### 输入
`$ARGUMENTS` — 要检查的设计文档路径（Markdown 或纯文本）

### Phase 1：命题抽取

读取完整文档，按 Section 逐一提取所有**原子化设计命题**，编号 P1、P2、...：

- 每条命题必须是**可验证的最小声明**（一个接口契约、一个字段约束、一个行为规则）
- 跨 Section 引用的相同概念独立提取（后续矛盾检查需要它们编号不同）
- 命题格式：`**Pn**: <主体> <动词> <约束>（来源：Section X）`

目标命题数量：每 500 字约 5–10 条，不设上限。

### Phase 2：命题间矛盾与遗漏检查

对所有命题两两比对（重点检查跨 Section 的）：

**矛盾**：两条命题对同一主体有相互否定的声明
- 格式：`### 矛盾 N: Px vs Py`，说明冲突内容，标注严重程度

**遗漏**：某命题声明了 A，但 A 依赖的 B 在文档中没有命题覆盖
- 格式：`### 遗漏 N: Px 依赖未定义的 <B>`，说明影响

严重程度判定：
- 高：会导致实现时运行错误或数据丢失
- 中：会导致行为不确定或测试无法通过
- 低：文档不清晰，不影响功能

### Phase 3：设计点交叉覆盖矩阵

从命题中归纳 **8–20 个核心设计点**（D1–DN），每个设计点对应一组相关命题：

| 编号 | 设计点 |
|------|--------|
| D1 | ... |

然后对所有设计点两两组合，检查"A 与 B 的交互是否有命题覆盖"：

| A | B | 覆盖？ | 位置/备注 |
|---|---|--------|----------|
| D1 | D2 | ✓ | Section X |
| D1 | D3 | **✗** | 说明为何这个交叉点重要、遗漏了什么 |

重点标记所有 **✗** 行，说明该交叉点的设计空白会引发什么问题。

### Phase 4：总结

分两块输出：

**关键问题**（需在实现前解决）：
- 按严重度排列，每条：问题描述 + 建议修复方向

**设计亮点**（可选）：
- 逻辑自洽、设计优雅的地方，一两句话

---

## 输出文件

报告写入 `<文档同目录>/<文档名>_finegrained_report.md`。

**写作流程**（防止 context 耗尽）：
1. 先用 Write 写骨架（四节标题 + TODO 占位符）
2. 逐节用 Edit 填充，每节完成再填下一节

## 汇报

完成后输出：
- 总命题数、矛盾数、遗漏数、矩阵空洞数
- 高严重度问题列表（一句话每条）
