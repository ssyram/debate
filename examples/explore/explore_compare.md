  1. **三方案对比表**（表格）：覆盖广度/命题清晰度/深度追溯/精度终止/用户可见性/上下文隔离/
     工具能力上限/配置维护成本 × 方案 A/B/C

  2. **决策矩阵**（表格）：场景特征 → 推荐方案 → 核心理由 → 失败边界；必须给出明确的场景→方案
     映射，不得「视情况而定」

  3. **明确推荐**（文字）：
     - 通用默认推荐是哪个方案，理由是什么
     - 方案 C 在什么具体条件下才真正优于另外两个方案
     - 每种方案最不适合的场景

  裁判有权综合辩手论点，但最终必须站队——给出一个对大多数实际探索任务最合适的方案推荐。

# 探索代理三架构比较：哪个方案最适合实际文献/代码情报探索任务？

## 三种方案简述

### 方案 A：SKILL 内联执行

在主会话中直接执行，用户全程实时可见。工具权限通过 allowed-tools 直接继承，无需额外配置。
状态通过 `.explore_state.json` 持久化，支持中断续跑。上下文随探索深入线性增长，遇到
paywall/JS 渲染/私有仓库时只能标记 Gap 并跳过。

### 方案 B：Agent 子代理

通过 Task 工具 fork 独立子进程，独立上下文隔离，主会话仅看最终返回值。上下文不污染主会话，
可并行为多个方向启动独立子 Agent。工具权限需 Task 调用时显式指定（已知 bug）。执行期间对用户
完全透明性为零。

### 方案 C：OpenCode 混合

Claude 作为编排引擎，外部 OpenCode session 作为执行节点。OpenCode 可直接 clone repo、读本地
文件系统、追踪函数调用链——这是方案 A/B 难以复制的能力。通信通过 `opencode_proxy.py` 以自然
语言消息交换（OpenAI chat completions 格式），不原生支持结构化 JSON 指令。需维护额外 proxy
进程，通信延迟叠加，两侧状态同步需额外设计。

## 三种方案定义（技术事实，不作为辩论内容）

**方案 A（SKILL 内联执行）**：
- 在主会话中直接执行（`.claude/skills/explore.md`），用户实时可见整个探索过程
- 工具权限通过 allowed-tools 直接继承，无需额外配置
- 状态文件 `.explore_state.json`，原子写入（tmp + mv），jq 校验，备份机制
- 约束：上下文随探索深入线性增长；中断恢复只能靠状态文件续跑
- 前两轮辩论裁定：「结构化探索的工程纪律，不是认知框架的忠实实现」

**方案 B（Agent 子代理）**：
- 通过 Task 工具 fork 独立子进程，拥有独立上下文和系统 prompt，主会话仅看返回值
- 工具权限继承有已知 bug，需在 Task 调用时显式指定工具列表
- 核心优势：上下文隔离（失败不污染主会话）、可并行执行多个主题子 Agent
- 核心劣势：执行期间黑箱，用户无实时可见性

**方案 C（OpenCode 混合）**：
- Claude 作为编排引擎（方向规划、命题编译、终止聚合），外部 OpenCode session 作为执行节点（实际搜索、抓取、代码阅读）
- 技术基础：`opencode_proxy.py` 将 OpenCode session 包装为 OpenAI-compatible 接口
- `opencode_proxy.py` 只支持自然语言消息交换（OpenAI chat completions 格式），不原生支持结构化 JSON 指令；如需传结构化数据，JSON 必须编码进 content 字符串
- 需维护额外 proxy 进程；通信延迟叠加；两侧状态同步需额外协议设计
- 可直接 clone repo、读本地文件系统、访问 OpenCode 原生工具——这是方案 A/B 难以复制的能力

## 可参考辩论张力（非必要，可以参考这些维度实现评判）

### 方案 C 的接口是否足够可靠？

自然语言消息交换是方案 C 的唯一接口。「去搜 Mamba 的原始论文并提取参数量和训练细节」这样的
指令，OpenCode 能否可靠执行并返回 Claude 可解析的内容？当 OpenCode 的返回格式不符合预期时，
Claude 如何感知和降级？这个接口的实际可靠性直接决定方案 C 是否成立。

### 深度能力是否值得协调成本？

方案 C 相对方案 A 的核心优势是 OpenCode 可以 clone repo、读本地文件。但这个优势只在需要深度
代码情报的任务中才体现。对于大多数文献调研任务，这个优势不存在，而协调成本始终存在。方案 C
的适用范围因此比看起来要窄。

### 实时可见性和上下文隔离能否兼得？

方案 A 有实时可见性但上下文随任务增长；方案 B 有干净隔离但黑箱执行；方案 C 理论上可以两者
兼顾（Claude 侧有可见性，OpenCode 侧有隔离），但实际上 Claude 看到的只是 OpenCode 的自然语言
返回，并非真正的实时过程可见性。这个张力在三方案比较中是真实的，不是表象。

## 核心例子：一次成功的 “错误归因分析体系” 调研

**本例子是一切讨论的出发点和参照物**，三方方案须以此为参照，分析实现本例子时的实际效果差异和到底能实现和如何超越本例子中的能力。

> 场景：用户指导下完成的 “错误归因分析体系” 调研，并最终输出了可用的参数化体系文件。

### 五阶段探索流程（已验证）

| 阶段 | 名称 | 输入 | 核心方法 | 输出品 |
|------|------|------|---------|--------|
| 1 | 全景扫描 | 领域范围描述（一句话） | 穷举性 BFS 搜索积累素材，对每个线头双向追踪引用，DFS 串联形成主题线索 | 7条主线、全景调研报告（38+ 体系、70+ 论文、时间跨度 1977-2026） |
| 2 | 信息组织与精简 | 原始条目（226 条） | 9 种移除标准 + Grounded Theory 常数比较法合并 | 精简条目（75 条）+ 两级体系初版 |
| 3 | 体系精化与参数化 | 初版体系（75 条） | 参数三分法（enum/enum_open/text）+ 边界判定指南（8 对混淆条目） | 参数化体系（v0.4，2673 行）|
| 4 | 多轮独立审查 | 参数化体系文件 | 3 轮 × 8 Agent 并行，三问法（互斥性/完备性/聚类能力），共识裁决 | 修改清单（350 行）+ 验证通过报告 |
| 5 | 跨 session 接力 | 当前阶段完成品 | 9 维度接力摘要（意图/概念/文件/错误/方案/用户发言/待办/当前/下一步） | 下一 session 的即开即用续接状态 |

**阶段转换判据**：每阶段由"完成信号"驱动，而非时间。例如：阶段 1 完成信号 = "BFS 穷举收敛：连续搜索无新概念/体系出现"；阶段 4 完成信号 = "数量守恒验证通过（修改数等于统计数）"。

### 搜索策略：BFS 穷举 → DFS 串联（已验证）

全景扫描阶段的实际搜索方法——注意：搜索路径是动态发现的，不是预先规划的。

**第一步：BFS 穷举积累**
不预设范围，见到任何可能相关的内容都记录（论文、技术帖子、综述、数据集报告、框架文档等）。
搜索触发词从领域描述中提取；每出现新概念立即追加为新的搜索词。

**第二步：双向引用追踪**
对每个线头（新概念、重要文献）执行：
- 向前追踪：该工作引用了哪些前驱？（历史脉络）
- 向后追踪：有谁引用了它？（后续发展）

**第三步：DFS 串联整理**
把积累的素材按相似主题串联，归纳成若干"主题线索"。
最终形成几条线索取决于领域结构，不是预设参数。
（参照案例中：BFS + DFS 后自然归纳出 7 条主题线索——这是结果，不是输入。）

**完成信号**：连续搜索 N 轮后再无新概念/体系出现，全景扫描收敛。

### 已验证有效的关键实践

**搜索与覆盖**：
- BFS 穷举积累 + 引用链双向延伸（向前 + 向后）确保了跨范式覆盖，不遗漏"隐蔽范式"
- 每出现新概念立即追加搜索词（锚点搜索），防止概念盲区
- DFS 串联整理把离散素材汇聚成可读的主题线索，线索数量由领域结构自然决定

**质量控制三问法**（最严格的审查方式）：
1. 能否构造一个实例同时落入多个分类？（互斥性测试）
2. 能否构造一个合理实例落入所有分类之外？（完备性测试）
3. 不同标注者面对同一实例，取值会"大差不差"吗？（聚类能力测试）

**已淘汰的做法**：
- 动态聚类（LLM 实时生成类别）→ 分类爆炸、一致性差，改为静态设计
- 宽泛标签（如"推理错误"）→ 无法指导具体行动，改为参数化细化
- Edit 工具逐行修改大文件 → 超时 kill，改为 Python 脚本 regex 批量替换
- "能跑就算通过"的验证口径 → 假阳性，改为数量守恒验证
