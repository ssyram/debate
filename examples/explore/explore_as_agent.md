     - 命题编译（TestableClaim 生成）在 Agent 私有上下文中是否更稳定？

  3. **深度维（D2）**：
     - Agent 可以获得与主会话不同的工具权限——explore Agent 应该有更多还是更少的工具？

# 探索代理：Agent 架构 vs SKILL 架构的设计选择

## 背景：已定型的 SKILL 设计（前一轮辩论结论）

以下设计要素在上一轮辩论（`explore_agent.md`）中已由裁判裁定，作为本轮的已知条件，无需重新辩论。

### 四象四阶段（MB/CC/D2/PA）

| 象 | 节点 | 已定型行为 |
|----|------|-----------|
| 广度象 | MB（Macro Breadth） | 生成 ≥6 个 Theme，覆盖 ≥4 种 source_class；内部轴 ≥3 个/经验层 ≥3 个；同源去重（seed_queries 重叠 >50% 合并） |
| 清晰象 | CC（Clarity Compiler） | 每 Theme ≥1 个 TestableClaim；falsifiable 字段含可观测量+比较基线+判定条件；locked 机制防漂移；不可证方向标为 REGULATIVE |
| 深度象 | D2（Evidence） | 来源类型决定强度基础分（PeerReviewed→STRONG，OfficialDocs/RepoCode→MODERATE，Discussion/Blog→WEAK）；access_status 显式记录；失败不脑补 |
| 精度象 | PA（Precision Aggregation） | 三重终止条件：覆盖率 ≥0.70 + TopK Jaccard ≥0.8（连续 2 epoch）+ 无阻塞 Gap；最多 5 个 epoch；超限输出"未收敛" |

### 已定型的状态文件方案

`.explore_state.json`，原子写入（先写 `.tmp` 再 `mv`），`jq -e .` 校验，每次写入前 `cp` 备份。如果文件损坏进入 `STATE_CORRUPT`：停止搜索，从备份恢复。

### 已定型的最小 CLI 依赖

仅需 `jq`。其余工具（WebSearch/WebFetch/Read/Grep/Glob/Bash）均为 Claude Code 内置，无需额外安装。

### 已定型的 Hooks 方案

- **PreToolUse**：检测 `.explore_state.json` 是否存在，仅做**警告**，非硬阻断（WebSearch 在非 explore 场景也会使用，硬阻断影响正常工作流）
- **PostToolUse**：每次 WebSearch/WebFetch 后做状态文件完整性校验（`jq -e .`），损坏时输出错误信息

### 裁判评述（前一轮）

> 四象在 SKILL 架构下不是「认知框架的忠实实现」，而是「结构化探索的工程纪律」。它的价值在于：让 LLM 的探索行为从不可审计的黑箱变为可检查、可中断、可复核的流程。

---

### 四象定义说明：概念层 vs 执行层

前一轮辩论在**概念层**定义了四象（MB/CC/D2/PA 各自负责什么认知任务、输出什么格式）。
本轮辩论在**执行层**讨论四象（在 Agent 架构下，同样的认知任务通过什么不同的技术机制实现）。

两个层次的关系：
- 概念层不变（广度象仍是"多源覆盖"，清晰象仍是"可证伪命题编译"，等等）
- 执行层因架构不同而不同（状态如何维护、工具如何隔离、终止如何传递）

round1_task 讨论的"四象在两种架构下的差异"是**执行层差异**，不是概念层差异。

## Agent 架构的关键差异

在 Claude Code 体系中，SKILL 和 Agent（sub-agent）是两种本质不同的执行模式：

### SKILL（当前已定型方案）

- 在主会话上下文中**内联执行**
- `allowed-tools` 约束工具集
- 状态与主会话**完全共享**：探索过程中所有工具调用、输出、状态变更都在主会话上下文中积累
- 每个 epoch 的输出**实时可见**，用户可随时中断
- 失败会**污染主会话**：错误信息、损坏状态均保留在上下文中

### Agent（sub-agent，通过 Task 工具）

- 通过 `Task` 工具 **fork 出独立子进程**，有独立上下文
- 可设置独立的 system prompt，与主会话完全解耦
- 与主会话的通信**仅通过返回值**（Task 工具的返回字符串）
- 可配置独立工具权限（可与主会话不同）
- 执行期间是**黑箱**——主会话看不到中间过程，直到 Task 返回
- 失败有**明确的错误返回**，不污染主会话上下文
- 多个 Task 可以**并行执行**（为不同 Theme fork 不同子 Agent）

### Agent 引入的新能力

| 能力 | SKILL 能否实现 | Agent 实现方式 |
|------|--------------|--------------|
| 上下文隔离 | 否（共享主会话） | 每个 Task fork 独立上下文 |
| 并行执行 | 否（串行） | 多个 Task 并发 |
| 独立工具权限 | 有限（受主会话约束） | Task 可配置独立工具集 |
| 显式输入/输出边界 | 否（通过文件/上下文隐式传递） | Task 参数 + 返回值 |
| 显式失败边界 | 否（失败留在上下文） | Task 有明确错误返回 |

## 核心设计张力（辩论焦点）

### 1. 状态管理

SKILL 方案靠 `.explore_state.json` 共享文件实现跨 epoch 状态持久化，主会话上下文是这一机制的"噪声背景"。

Agent 方案可以把**所有状态完全内化在私有上下文中**——`.explore_state.json` 可以是 Agent 的内部格式，主会话只看最终返回值，不关心中间状态。

**问题**：哪种更干净？共享文件方案给主会话提供了实时可见性，但也引入了文件 I/O 的失败模式；私有上下文方案消除了文件竞争，但主会话失去了对探索过程的观测能力。

### 2. 并行能力

Agent 架构原生支持并行：可以为 MB 阶段生成的每个 Theme 单独 fork 一个子 Agent，所有 Theme 并发探索。

SKILL 是串行的：所有 Theme 必须顺序处理，总时间 = Σ(每个 Theme 的探索时间)。

**问题**：并行化是否值得引入的协调复杂度？并行 Agent 如何汇总结果？主 Agent 如何合并多个子 Agent 的 `.explore_state.json`（如果有的话）？如何避免子 Agent 之间的重复搜索？

### 3. 工具权限

SKILL 使用主会话的工具集（通过 `allowed-tools` 约束）。Agent 可以获得与主会话完全不同的工具集。

**问题**：explore Agent 应该比主会话有更大还是更小的权限？
- 更大：允许访问特殊学术 API、私有代码库（D2 深度取证的天然扩展）
- 更小：只读约束（Glob/Grep/Read/WebSearch/WebFetch），防止意外副作用——比 PreToolUse hook 更强的硬约束
- 相同：最简单，无需额外配置

### 4. 交互粒度

SKILL 每个 epoch 实时输出状态报告，用户可随时中断并看到当前进度。

Agent 是"黑箱"直到返回——用户发出 Task 调用后，直到 Agent 返回前看不到任何中间状态。

**问题**：如何在 Agent 的上下文隔离（隔离带来的可靠性）和 SKILL 的实时可见性（用户体验）之间取得平衡？中间状态写入共享文件 + 主会话轮询是否是合理的妥协？

### 5. 失败语义

SKILL 失败（状态文件损坏、工具调用失败、LLM 胡言乱语）会留在主会话上下文中，后续对话会看到这些失败的痕迹。

Agent 失败有明确的错误返回（Task 工具返回错误字符串），主会话上下文保持干净。

**问题**：这是否改变了 Gap 和终止条件的设计？如果 Agent 可以干净地失败并重启，是否应该放宽 Gap 的 `blocking` 判定——允许 Agent 在遇到阻塞 Gap 时直接失败（而非降级），由主会话决定是否重试？

## 典型使用场景（Agent 视角）

同样的三个场景，这次考虑 Agent 架构下的用户体验和系统行为差异：

### 例 1：调查 Mamba（状态空间模型）架构

**SKILL 模式**：主会话开始探索，实时输出每个 epoch 的覆盖表和 TopK Claims。用户可以在 epoch 2 时说「先停下，重点看 inference 性能对比」，SKILL 即时响应。

**Agent 模式**：主会话 fork 一个 explore Agent（或为每个 Theme fork 独立子 Agent），等待返回。用户在等待期间可以继续使用主会话做其他工作。Agent 返回结构化报告（JSON + Markdown），主会话基于返回值决定是否追加探索。

**问题**：Mamba 调研通常需要 5-8 个 Theme，耗时较长。Agent 的并行执行是否显著缩短总时间？黑箱等待是否让用户体验更差？

### 例 2：LLM 代码补全开源方案调研

**SKILL 模式**：探索过程与主会话交织，用户可以随时插入追问（「StarCoder 的 context window 是多少？」），SKILL 立即响应。

**Agent 模式**：代码补全方案对比是结构化问题，输入/输出边界清晰——适合 Agent 返回结构化 JSON 供主会话进一步处理（如生成对比表格、启动后续辩论）。

**问题**：当探索结果需要被主会话进一步处理（而非直接呈现给用户）时，Agent 的结构化返回值是否比 SKILL 的上下文内联输出更有价值？

### 例 3：AI Agent 的 memory 机制调研

**SKILL 模式**：memory 机制本身是一个宽泛话题，探索过程中 MB 可能动态扩展到 10+ Theme，上下文可能超长。

**Agent 模式**：可以为每个子主题（短期记忆/长期记忆/外部存储/向量数据库/图数据库）单独 fork Agent，并行探索，最后主 Agent 合并结果。上下文隔离防止了主会话的长度爆炸。

**问题**：当探索范围不可预知（宽泛话题、动态发现的子主题数量不固定）时，哪种架构的上下文管理更优？
