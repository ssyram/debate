middle_task: |
  ## 中间轮：具体设计决策

  基于第一轮的可行性分析，针对以下四个设计决策给出明确立场：

  ### 决策 A：SKILL Prompt 的抽象层次

  两种极端：
  - **高度抽象**：用四象协议描述认知框架，让 LLM 自主解释执行（高灵活性，低可预测性）
  - **步骤化**：显式列出"阶段 1: 清晰化目标 → 阶段 2: 广度扫描 → ..."（高可预测，低灵活）

  哪种在实际文献情报场景中产生更可靠、更可重复的行为？
  是否存在一个混合点：抽象原则 + 具体检查点？

  ### 决策 B：Hooks 配置（Claude Code 主平台）

  对以下每个 hook 类型，给出明确的「值得做 / 不值得做」判断 + 理由：
  - **PreToolUse**：限制 explore SKILL 只能调用只读工具（Glob/Grep/Read/WebSearch/WebFetch）
  - **PostToolUse on WebSearch/WebFetch**：记录已访问 URL，避免重复探索
  - **Stop hook**：在探索结束前强制输出结构化摘要
  - **UserPromptSubmit**：在用户提交后注入四象协议上下文

  ### 决策 C：MCP 服务器推荐

  针对计算机科学情报/文献场景，给出具体 MCP 服务器推荐：
  - arXiv / 学术论文搜索
  - GitHub 代码搜索
  - 技术文档搜索
  - 通用 Web 搜索增强

  对每个推荐项说明：对应哪个四象维度，安装优先级（必装 / 推荐 / 可选），以及它解决了
  什么工具集的哪个具体缺口。

  ### 决策 D：子代理 vs 内联执行

  - **子代理**（fork sub-agent，隔离上下文）：避免污染主会话，但增加调度开销
  - **内联 SKILL**（主会话直接执行）：简单，但探索过程的大量输出会填满上下文

  这个选择对四象的哪个维度影响最大？文献情报场景应选哪个？

    Linus 的批判中哪些是真实的过度设计风险
  - **认识论清醒**（20%）：设计是否正视了 SKILL 与真正认知框架之间的差距；
    康德的批判中哪些边界在设计中需要明确标注

# 四象探索代理：计算机情报 SKILL 设计辩论

## 辩论背景

本次辩论以 **v3 四象认知引擎**为设计参照，探讨如何将这套认知框架的哲学精髓移植为一个实用的 Claude Code SKILL——一个以计算机科学情报和文献为核心场景的 Explore Agent。

## v3 四象认知引擎简述

v3 引擎采用两层分离架构（Layer 1 问题级 + Layer 2 命题级），四象节点协同工作：

**节点说明**："四象"指四个核心节点（MB/CC/D2/PA），是认知引擎的主体。Layer 1 完整流水线还有 QN（问题规范化）、RB（修复）、AS（答案组装）三个辅助节点；本次辩论聚焦四象核心节点的移植，辅助节点不在讨论范围内。

| 象 | 节点 | 核心职责 |
|----|------|---------|
| 广度象 | MB（Macro Breadth） | 从多类张力源（内部轴分裂、利益方冲突、外部立场）系统性生成探索方向，内部配额分配（内部轴 50% / 经验层 50%），同源性去重 |
| 清晰象 | CC（Clarity Compiler） | 将模糊 HypothesisDraft 编译为带 falsifiable_statement 的 TestableClaim，无法精确化则路由为 RegulativeIdea |
| 深度象 | D2（Layer 2 Evidence） | 针对精确命题收集 EvidenceAtom（单轴绑定，强度分级），按轴加权评分，构建置信链 |
| 精度象 | PA（Precision Aggregation） | 追踪轴覆盖率（阈值 0.70）与排名稳定性（连续 2 epoch），满足三条件后终止 |

四节点协同流程：MB → CC → D2 → PA；PA 发现缺口 → Repair → 循环收敛。

**关键约束**：完整的 v3 引擎依赖强类型状态机、持久化 EngineSnapshot、外部 EvidenceRef 注入。本次设计的约束是：仅有 LLM + 有限工具集，无持久状态存储。

## 关键概念速查

- **Epoch**：一次完整的 MB→CC→D2→PA 周期；"连续两 epoch 排名稳定"指连续两次周期后排名前 K 项无显著变化

- **GapSpec**：系统发现的"未覆盖缺口"的结构化描述，包含缺口类型（如高权轴无覆盖）和是否阻塞终止

- **EvidenceAtom**：单条证据的最小单元，绑定单一评估轴，含极性（支持/反对）和强度等级

- **覆盖率**：已有证据覆盖的评估轴权重之和 / 所有轴权重之和，阈值 0.70

- **四象协议**：在本辩论中指将四个维度（广度/清晰/深度/精度）编码为 SKILL 行为指令的方案；尚待辩论确定具体形式

- **同源性**：两个探索方向如出自同一视角/前提，则为同源；v3 中用 Jaccard 相似度 ≥ 0.6 作为数值判据

- **轴覆盖率**：见上方"覆盖率"定义

## 设计目标

将四象抽象为 Explore Agent 的四个操作维度：

- **广度维**：系统覆盖多信息来源（arXiv、GitHub、技术博客、官方文档、代码库），识别不同角度，不只是顺序检索
- **清晰维**：在大量搜索开始前，将用户的模糊探索意图分解为精确的可执行查询集，定义「找到了」的判据
- **深度维**：在初步发现后，追溯引用链、查看源码实现、验证关键声明，不停留在摘要/标题层
- **精度维**：持续评估探索完整性，达到覆盖阈值后收敛并结构化输出，而非无限漫游

## 典型使用场景

Explore Agent 的目标用户场景：用户只需一句话描述，Agent 负责从描述中自主规划探索路径，无需预先知道任何搜索词。

- **例1**：用户说「帮我调查 Mamba（状态空间模型）架构」，期望输出：核心论文摘要、主流实现对比、应用领域分布、与 Transformer 的关键差异

- **例2**：用户说「我要用 LLM 做代码补全，有哪些开源方案」，期望输出：主流方案（DeepSeek Coder / StarCoder / Codex）对比、关键指标、部署要求

- **例3**：用户说「调研 AI Agent 的 memory 机制」，期望输出：主流记忆架构分类（短期/长期/外部存储）、代表系统对比、工程落地难点

## 已验证成功经验：系统性调查体系参考

> 以下来自一次真实的、完整执行的系统性领域调查项目（LLM 数学推理错误分类体系构建），
> 该体系经过多 session 迭代验证，可作为本次 Explore Agent 设计的实证参照。

### 五阶段探索流程（已验证）

| 阶段 | 名称 | 输入 | 核心方法 | 输出品 |
|------|------|------|---------|--------|
| 1 | 全景扫描 | 领域范围描述（一句话） | 穷举性 BFS 搜索积累素材，对每个线头双向追踪引用，DFS 串联形成主题线索 | 7条主线、全景调研报告（38+ 体系、70+ 论文、时间跨度 1977-2026） |
| 2 | 信息组织与精简 | 原始条目（226 条） | 9 种移除标准 + Grounded Theory 常数比较法合并 | 精简条目（75 条）+ 两级体系初版 |
| 3 | 体系精化与参数化 | 初版体系（75 条） | 参数三分法（enum/enum_open/text）+ 边界判定指南（8 对混淆条目） | 参数化体系（v0.4，2673 行）|
| 4 | 多轮独立审查 | 参数化体系文件 | 3 轮 × 8 Agent 并行，三问法（互斥性/完备性/聚类能力），共识裁决 | 修改清单（350 行）+ 验证通过报告 |
| 5 | 跨 session 接力 | 当前阶段完成品 | 9 维度接力摘要（意图/概念/文件/错误/方案/用户发言/待办/当前/下一步） | 下一 session 的即开即用续接状态 |

**阶段转换判据**：每阶段由"完成信号"驱动，而非时间。例如：阶段 1 完成信号 = "BFS 穷举收敛：连续搜索无新概念/体系出现"；阶段 4 完成信号 = "数量守恒验证通过（修改数等于统计数）"。

### 搜索策略：BFS 穷举 → DFS 串联（已验证）

全景扫描阶段的实际搜索方法——注意：搜索路径是动态发现的，不是预先规划的。

**第一步：BFS 穷举积累**
不预设范围，见到任何可能相关的内容都记录（论文、技术帖子、综述、数据集报告、框架文档等）。
搜索触发词从领域描述中提取；每出现新概念立即追加为新的搜索词。

**第二步：双向引用追踪**
对每个线头（新概念、重要文献）执行：
- 向前追踪：该工作引用了哪些前驱？（历史脉络）
- 向后追踪：有谁引用了它？（后续发展）

**第三步：DFS 串联整理**
把积累的素材按相似主题串联，归纳成若干"主题线索"。
最终形成几条线索取决于领域结构，不是预设参数。
（参照案例中：BFS + DFS 后自然归纳出 7 条主题线索——这是结果，不是输入。）

**完成信号**：连续搜索 N 轮后再无新概念/体系出现，全景扫描收敛。

### 已验证有效的关键实践

**搜索与覆盖**：
- BFS 穷举积累 + 引用链双向延伸（向前 + 向后）确保了跨范式覆盖，不遗漏"隐蔽范式"
- 每出现新概念立即追加搜索词（锚点搜索），防止概念盲区
- DFS 串联整理把离散素材汇聚成可读的主题线索，线索数量由领域结构自然决定

**质量控制三问法**（最严格的审查方式）：
1. 能否构造一个实例同时落入多个分类？（互斥性测试）
2. 能否构造一个合理实例落入所有分类之外？（完备性测试）
3. 不同标注者面对同一实例，取值会"大差不差"吗？（聚类能力测试）

**已淘汰的做法**：
- 动态聚类（LLM 实时生成类别）→ 分类爆炸、一致性差，改为静态设计
- 宽泛标签（如"推理错误"）→ 无法指导具体行动，改为参数化细化
- Edit 工具逐行修改大文件 → 超时 kill，改为 Python 脚本 regex 批量替换
- "能跑就算通过"的验证口径 → 假阳性，改为数量守恒验证

### 输出整合方式（输出文档器）

成功体系的输出不是一个文件，而是三层汇聚：

```
阶段中间产物（每阶段独立保存）
    ↓
全景调研报告（综合文献覆盖视角）     ──→ 综合输出文档
    ↓                                      （演示/汇报/交付物）
体系文件版本历史（v0.1 → v0.4）      ──→ 跨 session 状态文件
    ↓                                      （接力摘要，9 维度）
修改清单 + 验证报告（质量证据链）    ──→ 量化进展数据
```

"全面介绍输出文档器"的职责：整合各阶段产物，让从未参与过程的读者能从一份文档中理解：调查了什么（覆盖范围）、发现了什么（体系结构）、如何验证的（质量证据）、可以怎么用（使用指南）。

### 对本次 Explore Agent 设计的启示

1. **输入应为纯领域描述**：用户无需了解搜索词、数据库名称或方法论，Agent 负责从一句话描述中自主展开穷举性 BFS 搜索，逐步发现并串联主题线索
2. **输出应为多层次产物**：不只是一个报告，而是：全景扫描层（覆盖地图）+ 核心发现层（精简结构）+ 质量验证层（互斥/完备/聚类检验结果）+ 使用指南层（如何延伸和复用）
3. **完成信号而非时间**：Explore Agent 的"精度维"终止判断可以借鉴"完成信号"机制——每个维度设计可观察的完成信号，而不是"感觉差不多了"
4. **成功经验的可移植性**：BFS 穷举 → DFS 串联整理框架、三问法审查、9 维度接力摘要均已在多 session 验证，可直接作为 Explore Agent SKILL 的可复用动作序列

## 核心设计张力

1. 精度引擎的引入在何时？记得本 SKILL 以广泛调研为第一要务，不能因为追求精度而过早收敛，但是也不能完全不考虑精度导致漫无目的的搜索

2. 到底是否需要某种深度引擎？如果需要，需求点在哪里？记得本 SKILL 以广泛调研为第一要务

3. **抽象 vs 可操作的 prompt 设计矛盾**：高度抽象的四象协议赋予 LLM 灵活性但行为不可预测；步骤化可预测但丧失框架的认知价值。

4. **工具覆盖边界**：WebSearch + WebFetch + Grep 能否真正支撑深度维的追溯需求？系统性失效边界在哪里？需要什么额外的工具执行？

5. **认知 vs 检索的本质差异**：v3 四象是一个认知框架，而 LLM SKILL 本质上是指令遵循 + 工具调用序列。两者之间有多少是真正等价的，有多少是隐喻性借用？
