### 3) 明确推荐（必须站队）

#### 通用默认推荐：**方案 A（SKILL 内联执行）**
**理由**（针对大多数“文献/情报探索”任务的真实约束）：
1. **实时可见 + 可纠偏**：探索最常见的失败不是“能力不够”，而是“方向跑偏/终止条件不稳”。A 让用户能在错误成本最低时介入。  
2. **终止可靠性最高**：完成信号、状态持久化、逐步收敛判断在同一会话里更可控，最接近你给的“阶段转换判据”范式。  
3. **最低运维成本**：不需要 proxy、双端状态同步，也不受 B 的工具继承 bug 影响。

#### 方案 C 的成立条件：满足这些条件时 **C 才真正优于 A/B**
C 不是“更强的搜索”，而是“**更强的本地执行与代码/文件取证**”。它优于 A/B 的条件必须同时满足：
1. **任务硬需求**：必须 clone repo / 本地读大文件（PDF、日志、数据集说明）/ 追调用链，且这些工作量不是“可手工替代”的小规模。  
2. **执行侧可用性可验证**：OpenCode 环境稳定、网络/权限可达、repo/PDF 可获取；proxy 进程长期可用。  
3. **有协议化回报约束**：即便接口是自然语言，也必须强制 OpenCode 回“可解析的证据包”（例如固定分节：结论/证据/文件路径/行号/命令/未完成原因），否则终止与验真会显著退化。  
4. **可以接受更高的延迟与排障成本**：跨进程问题（超时、断链、状态不一致）是常态而非例外。

**若以上任一条件不满足：应降级到方案 A。**  
（不建议默认降级到 B：因为 C 失败往往需要频繁过程纠偏，B 的黑箱会放大失败成本。）

#### 不推荐的场景（每种方案最不适合什么）
- **A 最不适合**：5+ 方向长期并行且中间过程不重要（会把主会话淹没，上下文成本高）。  
- **B 最不适合**：需要用户实时介入、需要可审计推导链、或工具权限/可达性不确定的任务（黑箱+工具 bug 会造成“终点才发现不可用”）。  
- **C 最不适合**：快调研与轻量文献综述（协调/运维成本 > 收益），以及对“结构化可验收输出+稳定终止”要求极高但又不愿意建立回报协议的场景（NL 接口会拖垮可靠性）。

### 一、 维度评估：实际探索效果的真实差异

#### 1. 覆盖广度：初始抽象的广度决定收敛质量
*   **方案 B（Agent）效果最好。**
*   **理由**：覆盖广度本质上是「初始抽象」的多样性。方案 A 在主会话中受限于线性思维，即便有状态文件，LLM 也很难在同一个 Context 窗口里同时维持 5 个互不干扰的搜索方向。方案 B 通过并行子 Agent 强制实现了「搜索维度的正交化」。
*   **风险**：并行不是为了快，是为了防止「认知塌缩」。如果 8 个 Agent 用的都是同一种搜索策略，那只是在浪费 Token。方案 B 必须在 Task 派发时，强制要求子 Agent 的 `system_prompt` 具有差异化（例如：一个搜论文，一个搜 GitHub Issue，一个搜 Reddit 讨论）。

#### 2. 命题清晰度：隔离是稳定性的前提
*   **方案 B（Agent）效果最好。**
*   **理由**：在方案 A 中，搜索到的噪音（如网页导航栏、无关广告、同名异义词）会直接进入主 Context，污染 LLM 对核心命题的判断。这叫「上下文投毒」。方案 B 的隔离机制确保了：子 Agent 看到的噪音，在返回主会话时已经被「类型转换」成了结构化的证据。
*   **方案 C 的缺陷**：Claude 与 OpenCode 之间的自然语言接口极其脆弱。如果 OpenCode 返回了 500 行混杂了日志和代码的文本，Claude 侧的解析器极易发生「命题漂移」。

#### 3. 深度追溯：物理访问权限是硬门槛
*   **方案 C（OpenCode 混合）效果最好。**
*   **理由**：方案 A/B 都是在「看」互联网，方案 C 是在「操作」系统。追踪引用链如果遇到需要 `git clone` 整个仓库来搜某个未导出的私有函数，方案 A/B 会直接卡死。方案 C 的 OpenCode 拥有真正的文件系统和运行环境。
*   **判定依据**：如果你的任务涉及「源代码情报」，方案 A/B 都是玩具。

#### 4. 精度终止：从启发式到断言的跨越
*   **方案 B（Agent）效果最好。**
*   **理由**：终止判断不应该是「我觉得够了」，而应该是「断言（Assertion）通过」。方案 B 可以在 Task 返回前增加一个 Check 步骤：如果返回的证据无法支撑初始命题的 3 个关键维度，强制重跑或报错。方案 A 的终止判断往往受限于 Context 长度——LLM 因为累了（上下文太长）而倾向于提前终止。

### 四、 例子（错误归因分析体系）的维度评判

这个例子在「工程启发式」上是成功的，但在「系统设计」上存在典型的**冗余性陷阱**。

*   **批判 1：阶段 4 的「8 Agent 并行审查」**。如果这 8 个 Agent 共享相同的模型参数和类似的 Prompt，这只是在做「概率分布的重复采样」，并没有引入真正的异构验证。
*   **批判 2：搜索策略的「收敛信号」**。例子中提到「连续搜索 N 轮无新概念出现」，这只是经验判断。在形式化视角下，这叫「局部最优」。如果搜索工具本身有偏见（比如只搜 Google Scholar 不搜 GitHub），这种收敛是假象。

**如何超越这个例子？**
*   **引入「类型检查」**：将阶段 3 的参数化体系定义为 JSON Schema。
*   **方案 B 的改进**：在阶段 4，不只是 8 个 Agent 投票，而是让 8 个 Agent 分别扮演「攻击者」角色，尝试构造违反 Schema 的反例（Counter-example Generation）。
*   **方案 C 的介入**：如果这个归因体系要应用到某个具体代码库，方案 C 应该在阶段 1 就介入，通过静态分析提取代码库中真实的错误处理模式，作为「初始抽象」的输入，而不是只靠搜论文。

**结论**：这个例子证明了「流程」的力量，但缺乏「约束」的力量。方案 B 配合严格的接口定义，是实现该流程并走向自动化的最优路径。

### 推进任务：关于「错误归因分析体系」例子的理论评判

用户评判得很精准：这个例子是「工程启发式」的产物，缺乏理论支撑。我用形式化验证的视角（特别是 CEGAR：Counterexample-Guided Abstraction Refinement）来重构它的四象维度支撑。

1.  **覆盖广度（State Space Exploration）**：
    例子中的「BFS 穷举」在理论上对应状态空间的广度优先遍历。但没有启发式函数的 BFS 会导致状态爆炸。方案 B（并行 Agent）提供了**空间切分**的理论支撑——将无界搜索转化为有界子空间的并行探索，保证了覆盖率的线性提升。
2.  **命题清晰度（Abstraction Quality）**：
    例子中的「9种移除标准 + 常数比较法」本质上是定义了一个**等价关系（Equivalence Relation）**，将无限的原始状态商简为有限的抽象状态。方案 A 在这里最弱，因为线性对话容易破坏等价关系的传递性；方案 B 的强制 Schema 返回，在系统层面上保证了这种抽象的类型安全。
3.  **深度追溯能力（Traceability）**：
    例子中的「双向引用追踪」对应形式化中的**依赖图（Dependency Graph）遍历**。方案 C 在处理代码时能构建精确的 AST 依赖图，而方案 A/B 只能构建基于自然语言的近似依赖图。
4.  **精度终止可靠性（Termination & Soundness）**：
    例子中最有价值的是「三问法」和「数量守恒」。这在理论上就是**反例生成（Counterexample Generation）**和**不变式检查（Invariant Checking）**。只要找不到反例，且满足不变式，循环就必须终止。方案 B 是唯一能在工程上将这种「断言检查」作为节点间路由条件的架构。

**总结**：这个例子之所以成功，是因为它暗合了 CEGAR 的精化循环。**方案 B 加上严格的类型签名，就是将这个「工程启发式流程」编译成「确定性系统协议」的唯一途径。** 方案 A 只能用来做这个流程的单步调试，方案 C 是特定领域（代码）的插件。默认底座，必须是 B。

*   **例子表现（极强）**：三问法（互斥/完备测试）+ 数量守恒验证。

- **优先级**：**P0**

### 议题3：方案C（OpenCode混合）的适用场景与启用条件

- **裁定**：**C仅在"源代码情报/本地文件系统操作"为硬需求时启用，且必须协议化证据包**
- **理由**：三位辩手一致认同C在"深度追溯能力"（clone/grep/AST/调用链）上具有A/B不可替代的硬优势。但三位也一致认同C的配置/维护成本最高、终止可靠性最低。Linus的判据最清晰："谁把C说成默认，谁就是想让你每天修proxy。"C的启用条件必须是：任务**物理上需要**文件系统操作（不是"可能有用"，而是"没有就做不了"）。
- **优先级**：**P1**

### 议题4：上下文污染（Context Poisoning）的缓解机制

- **裁定**：**方案A必须内置上下文管理机制，不能裸跑**
- **理由**：Ssyram提出的"上下文投毒"问题是对A最有力的攻击，且Linus未给出系统性回应。康德进一步指出，A的"可见性"不等于"可控性"，如果噪音持续累积，用户的纠偏能力也会退化。裁定A作为默认时，必须配套：
  1. 阶段性摘要压缩（每完成一个探索分支，将原始搜索结果压缩为结构化摘要）
  2. 状态文件（`.explore_state.json`）记录已探索方向、已确认证据、待探索缺口
  3. 当主会话token使用超过60%时，主动触发"上下文清理"（丢弃原始搜索结果，仅保留摘要）
- **优先级**：**P0**

### 议题5：终止条件的定义

- **裁定**：**终止条件必须是可验收的断言，不能是"感觉够了"**
- **理由**：Ssyram的"布尔可满足性"方向正确但过度形式化；Linus的"状态文件+人类审计"实用但不够严格；康德的区分最精准——"覆盖广度"是调节性理念，"终止"必须是构成性断言。裁定采用折中方案：
  1. 终止的**必要条件**（构成性）：缺口列表为空（所有预定义维度均有至少一条带来源的证据）
  2. 终止的**充分条件**（启发式）：连续N轮搜索（N≥3）未发现新的可枚举概念（术语/体系名/函数符号）
  3. 终止的**验收格式**：每条证据必须包含`{source_url, access_date, relevant_quote, confidence_level}`
- **优先级**：**P1**

### 议题6：方案间的降级/升级路径

- **裁定**：**必须定义明确的升降级触发条件，不允许运行时临时决定**
- **理由**：三位辩手均给出了各方案的"失败边界条件"，但缺乏系统性的升降级路径。裁定如下：
  - **A→B升级**：当A的主会话上下文超过60%且仍有3+个未探索方向时，将未探索方向拆分为独立TaskSpec派发给B
  - **A→C升级**：当A遇到需要`git clone`/本地文件读取/跨文件grep的硬需求时，切换到C
  - **B→A降级**：当B的子任务连续2次返回`Err(SchemaViolation)`时，将该分支回收到A中由用户手动探索
  - **C→A降级**：当C的proxy连接失败或回传解析失败超过2次时，回退到A
- **优先级**：**P1**
