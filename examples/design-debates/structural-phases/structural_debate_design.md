### 问题背景

debate-tool 目前的辩论流程是固定的：每 N 轮交叉质询一次，最后由裁判总结。但是经过多次的实践应用，发现其实优质辩论和优质的问题推进其实有自己的范式，目前总结范式为：
- 现象分析：辩手先分析现有流程的现象具体表现
- 问题提取和确认：从现象中提取问题，确认这是一个真问题，确认问题的本质到底是什么？
- 问题分解：将原始的、模糊的用户问题分解为更具体的更具有可解答性的子问题，很多时候问题问对了解答自然就出来了，或者至少可以更简单地解答了
- 解答：针对子问题提出解决方案

### 背景：当前 debate-tool 的辩论流程

**现有机制**（客观描述）：

debate-tool 目前支持以下配置字段：
- `rounds: N` — 辩手循环发言 N 轮
- `cross_exam: M` — 前面 M 轮，每轮后一次交叉质询，即每位辩手找另一其中一位辩手提出问题质询
- `round1_task` — 第一轮的任务提示，一般是展开辩论用
- `middle_task` — 除了第一轮，每轮除了有可能存在的上一轮质询问题回答以外的软性推进讨论要求
- `final_task` — 最后一轮结束后运行，生成结构化收尾
- 裁判（judge）在所有轮次完成后发言，生成 `tag="summary"` 的最终裁定

**已观察到的使用模式**：

当辩题需要分阶段讨论时（如：先分析现象，再判断是否是真问题，再分解，再解答），目前的做法是：
1. 在 `middle_task` 里写阶段感知逻辑，让 LLM 判断当前阶段并推进
2. 在 `constraints` 里提示辩手"第一阶段尽快收敛"
3. 依赖 LLM 自觉遵守阶段边界

**潜在替代方案**：

### 方案一：middle_task 软推进（现状增强）

在 `round1_task` 和 `middle_task` 里写更精细的阶段感知逻辑，让 LLM 判断当前阶段并推进。不改动 runner.py，仅靠 prompt 工程实现阶段感知。只是一种 topic 的书写范式，完全不需要改代码。

**代价**：阶段边界依赖 LLM 判断，不稳定；无法硬性限制每阶段 rounds 数。

### 方案二：phases YAML

在 topic YAML 中增加 `phases` 字段，显式定义每个阶段的 rounds 数和 phase_prompt：

```yaml
phases:
  - name: 现象分析
    rounds: 2
    phase_prompt: "本阶段只描述现象，不评价"
  - name: 真问题判断
    rounds: 1
    phase_prompt: "判断是否构成真问题，尽快收敛"
  - name: 问题分解
    rounds: 2
    phase_prompt: "分解成具体子问题"
  - name: 解答
    rounds: 3
    phase_prompt: "针对子问题提出方案"
```

Runner 需要修改主循环来支持 phases 迭代。

**代价**：需要修改 runner.py；phases 数量和 rounds 分配需要人工预判，可能不准确。

### 方案三：辩题流（Debate Flow Pipeline）

把结构化阶段提升到**辩论之间**的编排层，而不是单次辩论内部。每个阶段是一场独立的 2r1x 辩论，跑到不动点后进入下一阶段：

```
初始辩题
  └─ 2r1x × N 至不动点（现象挖掘 + 问题判断收敛）
       ↓
  真问题分解
  └─ 2r1x × M 至不动点（输出 K 个子问题）
       ↓
  [并行] 子问题 1 … 子问题 K，各自跑到不动点
       ↓
  综合器（K 份 summary → 一份带索引的起手 summary）
```

**不动点判断**：每次 2r1x 结束后，裁判/final_task 输出约定格式的稳定性信号：
```
STABILITY: STABLE | NOT_STABLE
UNRESOLVED: [未解决的分歧点]
SUB_PROBLEMS: [若已分解，列出子问题]
```
Orchestrator 脚本 parse 这个信号决定是否续跑或进入下一阶段。长期分歧不阻塞，记录后直接推进已收敛内容。

**实现代价**：不需要改 runner.py。需要新写 orchestrator 脚本（约 200 行 Python），以及约定 final_task 的输出格式。

## 阶段引导

本场辩论建议按以下顺序推进（middle_task 会软性引导，不强制）：

1. **第一轮**：描述现象——你们在使用中观察到什么问题？这是真问题吗？问题的本质是什么？
2. **中间轮次**：比较三个方案，分别指出每个方案的潜在问题、实现成本、预期收益
3. **最后轮次**：综合推荐，给出可操作的第一步

### 议题

首先，本设计的哲学本身是否合理？分成大的四阶段是否是一个合理的结构？是否有更好的阶段划分方式？或者面对更多不同的话题是否应该应用不同的结构，而采用更好的结构是否通过 Phase YAML 能更灵活？但是如果从 “辩题流” 的角度看，通过高层编排，每一种编排成本更高，但是根本上更灵活，是否值得？

然后基于以上背景，讨论以下问题：现有的 round1_task 和 middle_task 软推进机制是否足以支撑结构化阶段辩论？还是需要实现方案二（phases YAML）或方案三（辩题流 Pipeline）？哪个方案最值得优先实现？
