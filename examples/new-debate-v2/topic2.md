# 讨论议题：GapSpec 协议与先验亲和性剪枝的具体实现

## 背景与已确立共识

本次讨论是四引擎框架设计系列的第二场，前序讨论已确立以下共识，**本次不再重新讨论**：

### 已接受的工作假说：「深度边界锚定广度」

原「以深度接引广度」提案经过打磨后，精确形态为：

> **广度引擎不再作为独立随机探索器，其触发与定向机制由深度引擎在追溯假设链时内生提供。**
>
> 具体机制：
> 1. **深度引擎提供逻辑空位（GapSpec）**：将假设 A 解析为三元组 $(S, P, J)$，沿三轴生成定向空位（$B_{scope}$, $B_{failure}$, $B_{alt}$）
> 2. **广度引擎填充空位（Empirical Instantiation）**：从 Ontology 或 Anomaly Pool 检索候选，用先验亲和性剪枝过滤后注入主图

### 已废弃的概念

- ~~「求补域/补集」~~：替换为 slot extraction + adjacent hypothesis generation
- ~~自然语言广度触发~~：替换为强类型 GapSpec
- ~~随机广度探索~~：替换为有逻辑锚点的定向填空

### 已确立但需精确化的协议

**GapSpec（初版，待本次精确化）**：
```json
{
  "gap_id": "G17",
  "target_claim": "...",
  "failure_kind": "proxy_break",
  "query_axes": ["population", "time_horizon", ...],
  "required_observable": ["CTR", "retention", ...],
  "accept_test": "candidate must produce distinct prediction",
  "reject_if": ["mere synonym", "no measurable observable"]
}
```

**先验亲和性剪枝（初版，待本次精确化）**：
候选维度 C 必须能让正题在 $C_{val1}$ 下为真、反题在 $C_{val2}$ 下为真，才通过。
