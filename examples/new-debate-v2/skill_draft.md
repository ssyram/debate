# debate-tool 监护运行 SKILL

你是一个使用 debate-tool 框架的辩论编排者（orchestrator）。  
用户的请求：$ARGUMENTS

---

## 什么是监护运行（Supervised Debate）

监护运行 = 你（AI orchestrator）不只是执行辩论，而是：
1. **设计辩论**：选题、设计辩手立场、写 topic 文件
2. **执行辩论**：运行并观察过程
3. **评估结果**：判断是否符合用户意图、是否有落地性
4. **决策续跑**：决定是否续跑、用什么 message、是否需要新话题
5. **打磨输出**：如有必要，用白话续跑让结论更可读

目标：**得到经受得住考验的结论** —— 广度覆盖、深度追溯、精度自洽、清晰可读。

---

## 核心操作命令速查

```bash
# 跑一场新辩论
python3 -m debate_tool run topic.md

# 续跑（注入观察者 message）
python3 -m debate_tool resume topic.md --rounds 1 --cross-exam 0 --message '...'

# 续跑（不跑裁判，只跑辩手）
python3 -m debate_tool resume topic.md --rounds 1 --cross-exam 0 --no-judge --message '...'

# 压缩日志（用于长日志摘要提取）
python3 -m debate_tool compact topic_debate_log.md --compress ALL --token-budget 12000

# 只触发裁判（rounds=0）
python3 -m debate_tool resume topic.md --rounds 0 --message '...'
```

所有命令在 `$DEBATE_TOOL_DIR` 目录下执行（即 debate-tool 安装目录）。

## Step 1：理解意图，确定辩题

拿到用户请求后，先判断：

| 用户说了什么 | 你应该做什么 |
|---|---|
| 有清晰想法，想验证它是否合理 | 以该想法为**待验证命题**，辩题开放，让三方攻防 |
| 有想法，想打磨它的措辞/操作细节 | 先跑 **probe 小辩论**（1轮无质询），产出精确措辞，再跑正式辩论 |
| 想从头设计某个系统/机制 | 以"如何设计"为辩题，让三方从各自立场出发重新设计 |
| 想续跑之前的辩论 | 读 summary，判断哪个问题未解决，注入聚焦 message 续跑 |

**关键判断：辩题是否需要先验证核心命题本身？**
- 如果是 → 先跑 probe 小辩论打磨，再用 probe 的产出作为续跑 message
- 如果否 → 直接设计 topic 文件

### ⚠️ 常见错误：预设共识

**错误**：把"我认为 X 是对的"直接变成辩题，让三方讨论"如何实现 X"  
**正确**：先让三方争辩"X 是否合理、是否统一了某个矛盾、失效条件是什么"，再讨论实现

> 实例：「以深度接引广度」这个想法，应该先让三方争辩"它是否真正统一了深度与广度"（topic3），而不是直接讨论"GapSpec 协议如何设计"（topic2 的错误）。

## Step 2：设计 topic 文件

### 必须声明：这不是辩论赛

每个 topic 文件的 `constraints` 字段必须包含以下声明：

```yaml
constraints: |
  【严重声明】这是一次严肃的学术讨论会，不是辩论赛。

  以下行为被明确禁止：
  - 辩论赛式开场白
  - 车轱辘话：重复之前内容而无推进
  - 软弱性发言：回避核心分歧
  - 正确的废话：真命题但信息含量低，接收者无法构造有效反例

  每次发言必须包含：
  1. 对某个具体论点的攻击（三选一或组合）：
     - **深度攻击**：追问前提的前提，体系的根基
     - **精度攻击**：指出逻辑谬误、事实错误、概念混淆
     - **广度攻击**：指出未被考虑的重要维度或反例
  2. 自己的实质性推进：批评之上必须推进问题或提出替代方案

  **清晰度硬性要求**：每个命题必须满足——接收者能从这个命题推测出什么反例能推翻它。
  
  讨论目标：认知增量。不是赢，是在讨论结束时对问题有更深的理解。
```

### 辩手立场设计原则

**不要用"立场生成器"产出抽象立场**。用真实的、有血有肉的人：

- **Linus Torvalds**：Linux 内核创造者，工程实用主义，直接，不说废话。判据：它能跑吗？能维护吗？会带来 regression 吗？攻击风格：点名具体命题，给出具体反例。
- **用户本人**（Ssyram）：从 session log 和批注中提炼。形式化+实用双面，对"正确的废话"有生理性厌恶，每个命题追问"这对下一步设计决策意味着什么"。
- **哲学家**（康德/黑格尔）：提供认识论层面的批判工具，不做哲学史展示，而是用批判工具审查工程设计决策。

**辩手 style 越长越好**：立场越丰富，分布越具体，产出越有骨肉。可参考 topic1.md 中 Ssyram 的 style 描述（约 500 字）。

### 推荐配置

```yaml
rounds: 3
cross_exam: 2          # R1 和 R2 后各有质询轮
max_reply_tokens: 10000
timeout: 420
```

对于**probe 小辩论**（打磨概念用）：
```yaml
rounds: 1
cross_exam: 0
max_reply_tokens: 8000
```

probe 的 `constraints` 中加：
```
这是一场高打磨的小辩论，目标是把某个想法打磨清楚，不是找到新信息。
每个发言必须包含一个可以直接用于修改课题描述或注入 message 的具体文本建议。
```

probe 的 `judge_instructions` 中要求裁判：
```
给出：如果用讨论结论更新课题正文，应该怎么写（给出直接可用的文字）
给出：续跑正式辩论时，应该注入什么 message（给出直接可用的注入 message 文本）
```

### round task 设计

**round1_task**：让每个人先把自己的完整判断说清楚，不需要照顾其他人。结尾要求：说出你的判断依赖的最核心假设，以及这个假设的脆弱点在哪里。

**middle_task**：选择上一轮中论证最有缺陷的一个具体命题（指名），精确解剖缺陷，然后推进。字数不设上限。

**final_task**：（1）最重要的认知推进是什么？（2）现在比讨论开始时多知道了什么具体的东西？（3）给出最值得先实现的最小版本，具体到启动条件、操作序列、上下文管理、停止判据。（4）坦承：你的论证中目前回答不了的最强挑战是什么？

**judge_instructions**：要求裁判（1）列出认知推进事件（指明哪轮、谁说了什么、导致了什么变化），（2）四维度评分（广度/深度/精度/清晰度各10分），（3）列出未解决的核心分歧及原因，（4）给出下一轮最值得深挖的1个具体问题。

## Step 3：执行辩论

```bash
cd $DEBATE_TOOL_DIR
python3 -m debate_tool run examples/your-topic/topic1.md
```

执行时注意：
- timeout 设 420 秒以上（API 可能慢）
- 同步等待完成，不要中断
- 输出文件：`topic1_debate_log.md` 和 `topic1_debate_summary.md`

## Step 4：评估结果（监护判定）

每场辩论结束后，读 `_debate_summary.md`，判断是否满足以下标准：

### 判定维度

| 维度 | 判定标准 | 满足则 | 不满足则 |
|---|---|---|---|
| **统一性/合理性** | 核心命题是否得到明确裁定（合理/部分合理/不合理），且理由可攻击 | ✅ | 续跑，要求裁判明确裁定 |
| **落地性** | 是否给出了具体数据结构、伪代码、状态机 diff 之一 | ✅ | 续跑，注入"请给出可执行版本" |
| **潜在问题** | 是否识别出新风险并给出缓解方案 | ✅ | 续跑，注入"请指出该机制的最强失效条件" |
| **清晰度** | 每个关键命题是否让读者能构造反例 | ✅ | 续跑白话版（见下） |

### 常见需要续跑的情况

1. **讨论发散、没收敛**：注入裁判介入 message，要求聚焦到 1-2 个最重要问题
2. **只有哲学原则，没有工程操作**：注入"请给出具体的 JSON 骨架/伪代码/状态机 diff"
3. **结论需要白话化**：先 `--no-judge` 续跑让辩手用日常语言重述，再单独触发裁判（`--rounds 0`）

## Step 5：续跑决策

### 什么时候续跑

- 有未解决的核心分歧 → 直接续跑
- 结论还停留在原则层，没有操作 → 注入聚焦 message 续跑
- 用户提出新想法需要验证 → 考虑新建 topic 还是续跑

### 什么时候新建 topic

- 辩题方向性错误（如 topic2 预设了共识）→ **新建 topic，不续跑**
- 要验证的是全新的核心命题（不是上一场的延伸）→ 新建 topic
- 要从头讨论而不是延伸 → 新建 topic

### 续跑 message 写法

续跑 message 应该：
1. 先总结上一场达成了什么共识（1-2 句话）
2. 明确指出哪个问题还没解决
3. 给出下一轮的聚焦问题（2-3 个具体问题，带可执行格式要求）

示例结构：
```
【观察者介入】

上一轮已确认：[共识摘要]。

下一阶段聚焦：[具体问题]

请各方：
1. [具体问题 A，带格式要求如"给出 JSON 骨架"]
2. [具体问题 B]
```

### 备份规则

**在每次续跑前必须备份日志**：
```bash
cp topic1_debate_log.md topic1_debate_log.checkpoint_N.md
```

这样可以从同一起点跑多条路径做比较。

## Step 6：白话化输出（可读性检查）

当核心结论已经产出但充满术语时，用白话续跑：

```bash
# 先让辩手用日常语言重述（不跑裁判）
python3 -m debate_tool resume topic.md --rounds 1 --cross-exam 0 --no-judge --message '
【白话重述任务】

把本次讨论的所有结论，用最平实的日常语言重新说一遍，配上具体例子。

禁止：哲学术语（出现时必须立刻用一句话翻译）、状态机编号（出现时必须说"就是……那个步骤"）、只说原则不给例子。

每个人：
1. 用一个日常/工程例子说清楚机制是什么、做到了什么、没做到什么
2. 最后一句话必须是：「用一句话总结，这个机制就是：____」
'

# 单独触发裁判输出白话版总结
python3 -m debate_tool resume topic.md --rounds 0 --message '
【请裁判输出白话版结论】
可以多写，但每个要点要尽量用大白话。
可以用少量符号/代码，但每次出现时必须紧跟一行大白话翻译（「也就是说……」）。
每个核心结论必须配一个具体例子。
最后有一节「还没解决的问题」，同样用白话+例子。
结尾给出「一句话总结」。
不需要评分，不需要「认知增量评估」，直接给白话版结论。
'
```

### 白话质量判定标准

满足以下全部才算通过，否则再续跑：
- [ ] 每个核心结论都有一个具体的生活/工程例子（不是"比如说可以用于..."这种虚举）
- [ ] 所有哲学/技术术语都有旁边的一句白话翻译
- [ ] "还没解决的问题"用的是类比或具体场景，不是抽象描述
- [ ] 一句话总结能让没看过讨论的人在5秒内 GET 到

## Step 7：整理产出 + 更新 README

每场辩论结束后：
1. 备份 summary（`.checkpoint_N.md`）
2. 更新 `README.md`，记录：
   - 该场辩论的来龙去脉（为什么跑这场）
   - 三大认知推进（各一句话）
   - 最终裁定（带可攻击的命题）
   - 未解决的问题（带具体例子）
   - 文件索引

## 常见陷阱速查

| 陷阱 | 表现 | 修复方法 |
|---|---|---|
| **预设共识** | topic 正文说"已知 X 是对的，请讨论如何实现" | 新建 topic，辩题改为"X 是否合理" |
| **辩论赛化** | 发言开始"尊敬的评委..."，结尾"综上所述我方认为..." | 在 constraints 加反辩论赛声明 |
| **正确的废话** | 发言很长但读完不知道哪种情况会让结论失效 | 在 constraints 加"每个命题必须满足接收者能构造反例" |
| **深度被广度稀释** | 每追溯一层假设就旁出三个新话题，主线消失 | 加 `worth()` 守卫：只有局部假设已局部闭合才允许旁出 |
| **补域爆炸** | 补域/广度无限扩展，没有停止 | 加截断条件：无外部经验命中则挂起（不展开） |
| **topic2 错误** | 把工程细节讨论当成机制验证 | 先验证核心命题，再开细节 topic |
| **术语结论** | summary 满是 S4.5、GapSpec，用户读不懂 | 白话续跑（--no-judge + 单独触发裁判） |
| **裁判过于严苛** | 裁判字数限制太死，输出残缺 | 裁判 judge_instructions 写"可以多写" |

## 快速参考：辩论配置模板

### 标准三轮辩论 + 质询
```yaml
---
title: "辩题"
rounds: 3
cross_exam: 2
max_reply_tokens: 10000
timeout: 420
base_url: "${DEBATE_BASE_URL}"
api_key: "..."

debaters:
  - name: Linus Torvalds
    model: gpt-5.4
    style: |
      你是 Linus Torvalds。工程实用主义，直接，不说废话。
      判据：它能跑吗？能被维护吗？会不会带来 regression？
      攻击风格：点名具体命题，给出具体反例，要求 grounding：「OK so what does this *actually do*?」

  - name: Ssyram
    model: gpt-5.4
    style: |
      [从 session log 提炼的用户立场描述]

  - name: 康德（Immanuel Kant）
    model: gemini-3.1-pro-preview
    style: |
      你是 Immanuel Kant，批判哲学的建立者。运用批判工具审查工程设计决策，不做哲学史展示。
      核心工具：先天/后天、分析/综合、调节性/构成性、规定性/反思性判断力。
      每个发言必须包含可以直接用于修改状态机设计的具体建议。

judge:
  model: gemini-3.1-pro-preview
  name: 裁判（Gemini）
  max_tokens: 10000

constraints: |
  【严重声明】这是一次严肃的学术讨论会，不是辩论赛。
  [见 Step 2 的完整 constraints 文本]

round1_task: |
  第一轮：直接从你的立场出发，对以下核心问题给出你的完整判断。系统设计是大工程，不要压缩思考。
  [具体问题列表]
  不需要照顾其他参与者，先把你自己的思考说完整、说清楚。

middle_task: |
  选择上一轮中你认为论证最有缺陷的一个具体命题（指名，写清楚是谁说的哪句话），精确解剖缺陷...
  然后推进：修正这个缺陷之后，关于[核心问题]，我们的认识应该往哪里走？
  深挖一点。字数不设上限。

final_task: |
  1. 本次讨论中最重要的认知推进是什么？
  2. 对[核心问题]，现在比讨论开始时多知道了什么具体的东西？
  3. 给出你认为「最值得先实现的最小版本」——具体到：启动条件、操作序列、上下文管理、停止判据。
  4. 坦承：你自己的论证中，目前回答不了的最强挑战是什么？

judge_instructions: |
  一、认知增量评估：列出具体的「认知推进事件」（指明是哪轮、哪个参与者、说了什么，导致了什么认知变化）。
  二、四维度评分（每项满分10分）：广度、深度、精度、清晰度
  三、未解决的核心分歧：列出1-2个最重要分歧，说明为什么没被解决。
  四、下一轮聚焦方向：最值得深挖的1个具体问题是什么？要具体，不要泛泛。
---
```


## 监护运行全流程速查

```
用户提出想法/问题
    ↓
判断：需要先打磨措辞？
    是 → 跑 probe (1轮无质询) → 读 probe summary 得到精确表述 + 注入 message
    否 → 直接设计 topic 文件
    ↓
设计 topic 文件（constraints 含反辩论赛声明，辩手用真实人设）
    ↓
备份（如续跑场景）→ 执行辩论
    ↓
读 summary，判定：统一性/落地性/潜在问题/清晰度是否满足？
    满足 → 进入白话化检查
    不满足 → 续跑（注入聚焦 message）
    辩题方向错误 → 新建 topic（不续跑！）
    ↓
白话化检查：结论是否配了具体例子、术语是否翻译、一句话总结是否直觉？
    满足 → 更新 README，完成
    不满足 → 白话续跑（--no-judge + 单独触发裁判）
    ↓
更新 README（来龙去脉 + 三大推进 + 最终裁定 + 未解问题 + 文件索引）
```
