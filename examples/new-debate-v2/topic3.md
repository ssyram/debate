# 讨论议题：「以深度接引广度」是否真正统一了深度与广度引擎？

## 背景

四引擎框架（广度/深度/精度/清晰度）的设计中，存在一个长期未解的问题：

**广度引擎的触发来源**：广度引擎最难实现的原因在于它没有天然的局部 stopping signal，而且触发来源存在争议——
- Linus/Ssyram 认为内部平台期（连续 N 轮无新攻击类型）可以触发广度
- 康德认为广度必须依赖外部「经验阻力」（Anomaly Pool 溢出），系统无法从内部完美一致中发现漏维

Ssyram 提出了第三条路：

## 新提案：「以深度接引广度」

**核心主张**：当深度引擎追溯假设链时，每找到一个底层假设 A，系统同时生成与 A 相关的广度探索空位。广度扩展因此成为深度追溯的自然副产品，不再需要独立的触发机制。

**经过 probe 打磨后的精确表述**：
- 「补域」不是数学补集，而是将假设 A 解析为 $(S, P, J)$ 三元组（作用域/性质/支撑），沿三轴生成定向空位：
  - $B_{scope}$：当前作用域外的未覆盖相邻子域（如「从新用户」推至「回访用户」）
  - $B_{failure}$：使当前性质 P 反转的条件（如「寻找点击率不代表满意度的场景」）
  - $B_{alt}$：支撑 J 不成立时的替代机制（如「寻找不依赖 CTR 的质量代理指标」）
- 深度引擎提供**逻辑空位**，广度引擎用外部经验（Ontology/Anomaly Pool）填充
- 若无外部经验可填，该方向自动截断

## 待验证的核心命题

1. **统一性命题**：这个机制是否真正「统一」了深度和广度，使两者成为同一追溯路径的两个方向？还是只是深度为广度提供了更好的 query，广度引擎本质没变？

2. **失效条件命题**：这个机制在什么情况下失效？（补域爆炸？深度被广度稀释？广度内容来源仍需外部注入？）

3. **状态机影响命题**：如果这个机制成立，原有的 S4(DepthProbe) → S5(BreadthProbe) 独立触发路径应该如何修改？是合并为同一引擎的两个阶段，还是保持独立但改变触发规则？

## 原四引擎状态机（供参考修改）

```
S0(Draft) → S1(Clarify) → S2(TestableClaim) → S3(PrecisionCheck)
S3 → S4(DepthProbe)    [矛盾源于隐含前提]
S3 → S5(BreadthProbe)  [矛盾表现为死锁]
S4 → S1                [挖出新假设，重过清晰度门控]
S4 → S7(Suspended)     [到达操作化极限]
S5 → S9(SchemaChallenge) [旧 schema 无法表达新对象]
S6(Verified) → S4      [外部反例命中]
```

问题：如果「以深度接引广度」成立，S4 在追溯假设时应该**同时**生成广度空位，这意味着 S4 不再只输出「新假设→回 S1」，还输出「广度空位→触发 S5」。这改变了 S4 和 S5 的关系。
