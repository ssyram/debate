## 大致设计

Prompt:
> 抽象来说，我们有广度引擎，负责：根据需求（第一轮） / 根据反馈（后面轮次），其实根据反馈也是根据需求，我们应该有个全貌综合器负责高精度维护全方案和全貌，然后通过广度引擎产生张力，中间通过一个分解引擎将 claim 和设计爬出来，作为一个个问题扔给深度引擎，深度引擎负责稳固，也就是逐个向前拆解其理由/原因，再根据其直觉性程度再迭代拆，拆到一个地方直觉程度足够高，我们可以将其作为原子公理的时候就保留下来（这些东西未来还要面对冲突检查，可能好几个东西之间是有冲突的，要么找到调和，从而进一步确定系统范围和/或打磨词义精确描述，要么产生分裂和分别的讨论，即不同情况下采取不同体系），精度引擎/清晰度引擎在深度引擎和全貌综合器中需要实时进行更高的掌控和检查，它专门负责找漏洞执行（逻辑）攻击，攻击纳入 worker 考虑范围，进一步思考和改进整个过程。如果深度引擎否定了方案，则需要回馈到综合器，综合器简单确信判断哪些地方存在问题，如果判断本身做不了，扔给深度引擎负责命题性确认，如果判断是可马上简单修复（替换原来已经有的合适部件方案，直接塞进去）就直接修复不用管，如果是发现有空缺 —— 导致某一整块都用不了了，直接反馈给广度引擎，重新根据需求生成。也就是维护器负责从头到尾维护我们需要的总输出的（一个或者多个的）全貌，维护器会发出描述性请求和命题性请求，分别发给广度引擎（较低精度素材引擎）和深度引擎（高精度命题证明/分解引擎），这中间精度引擎和清晰度引擎分别爬过维护器和深度引擎的部分，执行攻击检查，深度引擎负责命题递归分解，精度引擎则负责攻击这些分解步骤里面的论证性，强迫深度引擎做出回答，维护器执行维护的时候精度引擎则负责逐个部件敲打攻击，生成辅助性宏观命题，负责在维护器执行的每一步都提问，强迫维护器做深思 —— 但是维护器深思只有确信和不确信两种结果，不确信马上找深度引擎处理，确信则可以继续执行。这个方案如何？

直接结论
方向是对的，而且比 v3 更贴近“大系统设计”。  
但你现在这版角色切得有点过细，再往下会先死在控制流和职责重叠，不是死在能力不够。
---
这套方案里最对的地方
1. 有一个“全貌维护器”作为中心
   - 这是对的。
   - 它本质上就是 blackboard owner / state owner，不是普通 orchestrator。
2. 不是从 claim 出发，而是从全系统出发
   - 这点非常关键。
   - 大系统设计必须先有全局结构，再从里面爬出局部问题。
3. 先广度产张力，再分解成可深挖对象
   - 这也对。
   - 广度不是直接产结论，而是产 tension / candidate / unresolved area。
4. 深度引擎负责递归拆解理由，直到可接受原子
   - 这是正确方向。
   - 只是“原子公理”要谨慎，最好叫 accepted primitive / working axiom。
5. 攻击/挑错必须常驻
   - 也对。
   - 没有 adversarial pressure，设计很容易自嗨。
---
最大问题：你现在有点“过分拆角色”了
1) 精度引擎 / 攻击引擎 / 清晰度引擎 现在是重叠的
它们都在做一类事：
- 找漏洞
- 逼问
- 检查歧义
- 检查论证是否站得住
- 压迫维护器/深度引擎说清楚
这说明它们还不是三个独立 engine，  
而更像两种东西：
- 一种统一的 Critic / Challenger 能力
- 若干检查维度
  - 逻辑攻击
  - 术语澄清
  - 精度约束
2) 分解引擎 和 深度引擎边界也有点模糊
你现在写法里：
- 分解引擎：把 claim 和设计爬出来，扔给深度
- 深度引擎：递归拆理由直到原子
这两个其实是一条连续链：
> 对象化 / 编译 / 递归求证
所以更合理的边界通常是：
- 分解/编译：把全局问题转成可评估对象
- 深度/验证：对这些对象做递归论证、反驳、稳固
3) 如果攻击引擎独立成单独常驻调度者，会制造死循环
典型坏循环：
- 深度给结论
- 攻击引擎攻击
- 维护器不确信
- 再丢回深度
- 深度修一点
- 攻击继续攻击
- 永无止境
所以攻击最好不是“并列主引擎”，而是：
- 深度中的一个 phase
- 或维护器触发的审查模式
---
我建议的收敛版：3+1 架构
1. 维护器 / Maintainer（唯一全局状态拥有者）
职责：
- 持有完整 DesignState
- 决定当前缺什么
- 发两种请求：
  - 描述性请求
  - 命题性请求
- 应用修复、关闭问题、重新开洞
- 决定是否终止
这个角色必须保留，而且必须唯一。
---
2. 广度引擎 / Breadth
职责：
- 根据需求或缺口生成：
  - tensions
  - alternative structures
  - candidate modules
  - interface variants
  - risk hypotheses
  - missing regions
- 分两种模式：
  - 初始广度
  - repair 广度
它不是“结论引擎”，而是设计空间展开器。
---
3. 分解/编译引擎 / Decomposer-Compiler
职责：
- 从全貌和候选中爬出：
  - DesignQuestion
  - DesignAssertion
  - Conflict
  - OpenGap
  - InterfaceConstraint
  - InvariantCandidate
- 把大块描述转成可深挖对象
- 同时承担清晰度门控
  - 不清晰就拒收或回退
  - 不让烂对象进深度
你说的“清晰度引擎”，更适合并到这里。
---
4. 深度引擎 / Depth
职责：
- 对单个对象递归拆解
- 给出支持链 / 反对链
- 找前提
- 找反例
- 判断可否接受为 working axiom / accepted primitive
- 返回：
  - accepted
  - rejected
  - blocked
  - conflict
  - refine-needed
攻击/精度能力并到这里最合理。  
也就是深度引擎内部有：
- prove mode
- challenge mode
- contradiction mode
而不是再拆一个平级引擎。
---
所以你的“精度引擎”怎么处理？
不建议作为独立主引擎
更建议把它降级成 横切能力（cross-cutting capability）
分别挂在：
- 分解/编译阶段
  - 查术语是否含混
  - 查对象是否过大
- 深度阶段
  - 查论证是否跳步
  - 查前提是否偷换
- 维护阶段
  - 查全局是否自洽
  - 查 patch 是否引入新裂缝
也就是它不是独立 engine，  
而是一个统一的 critic policy / attack policy。
---
维护器该不该只输出“确信 / 不确信”？
不够。
二值太粗，会让很多正常状态都被错误升级。
建议至少用：
- confident_accept
- tentative_accept
- needs_depth_check
- needs_regeneration
- local_patch_available
- global_structure_broken
- blocked
这样维护器才真能“维护”，而不是只会转单。
---
这套架构最危险的地方：终止条件
你现在最大风险不是生成不出东西，  
而是永远还能继续想。
必须有硬停止：
- coverage 达标
- 核心结构连续 N 轮稳定
- 新攻击不再改变关键骨架
- breadth 新产物边际价值下降
- 深度递归达到 budget 上限
- unresolved gaps 降到阈值以下
- 或进入 budget_exhausted / schema_exhausted
没有这个，维护器会永远觉得“再想一下也许更好”。
---
一个更干净的控制流
初始化
需求 -> 维护器建立 DesignState
循环
1. 维护器检查缺口
2. 缺结构 → 调广度
3. 广度产候选/张力
4. 分解器把候选编译成对象
5. 深度引擎逐个递归验证/攻击
6. 返回 verdict / gap / conflict / patch
7. 维护器：
   - 局部可补 → 直接 patch
   - 整块失效 → 重新请求 breadth
   - 只是命题未定 → 再丢 depth
8. 检查终止
---
一句话评价你当前方案
不是错，而是“想法太全，角色太细”。  
真正该保留的是你的核心洞见：
- 全貌先于局部
- 局部从全貌中爬出
- 深度递归证明
- 攻击常驻施压
- 维护器统一收口
真正该删的是：
- 把攻击、精度、清晰度都拆成平级主引擎
---
最终建议
保留
- 全貌维护器
- 广度引擎
- 分解/编译
- 深度验证
- 攻击作为横切机制
合并
- 精度 + 攻击 + 清晰度
- 分解 + 清晰编译可紧耦合
不要做
- 6 个平级引擎互相发消息
- 多个角色共同拥有全局解释权
- 没有硬终止条件的持续审查
---
最短版结论
这方案核心方向是对的，但当前版本最需要的不是再补角色，而是压缩角色。  
我建议你把它收敛成：
> 1 个维护器 + 3 个主引擎（广度 / 分解编译 / 深度验证）+ 1 套横切攻击策略
这样才像一个能跑的 v4，而不是一个永远开会的委员会。
