> 2026-03-18T17:46:20.014748

# 第一部分：系统全景解释

## 这个系统是什么，解决什么问题

这个系统叫做"v4 认知引擎"，它要解决的问题是：**如何让大语言模型（LLM）处理复杂的、开放域的大问题——比如设计一个软件架构、评估一个庞大的代码库、分析一个历史事件的因果——并产出可靠的、结构化的结论，而不是一堆看似深刻实则空洞的废话。**

当前的做法（比如让多个LLM扮演不同角色进行"辩论"）有一个致命缺陷：LLM在对话中会迅速收敛到一种"防御性共识"——各方找到最安全的公共地面，然后互相附和，不再产生真正的新信息。这被称为"平凡劣化"。原因很简单：在固定的模型参数和提示词下，LLM的展开本质上是在解包已有的知识，不会凭空产生信息增量。让两个LLM互相回应，只会让它们更快地收敛到预训练分布中最常见的"安全答案"。

v4 认知引擎的核心设计思路是：**不要把LLM当成能思考的团队成员，把它当成一块不稳定但强大的计算硬件。在它外面套一层严格的、确定性的控制系统，用结构化的数据和明确的协议来约束它的行为，从它身上榨取有用的局部判断，同时用外部机制来保证全局的逻辑一致性。**

打个比方：LLM 是一台功率巨大但精度不稳定的机床。你不能让它自己决定加工什么、怎么加工。你需要一个精密的数控系统（维护器）来下达指令，一套严格的量具（攻击引擎和外部预言机）来检验成品，以及一套明确的工艺流程（协议层）来确保每一步都可追溯、可审计。

## 各组件的职责和相互关系

系统由五个核心组件和一个协议层构成。

### 1. 中央维护器（Maintainer）——调度中枢

维护器是整个系统的"操作系统内核"。它是一段**完全不调用LLM的确定性代码**——一个 while 循环加上一些 if-else 分支。

它的唯一职责是：**维护一棵全局的命题树（DAG），并根据树中节点的状态，决定下一步把哪个节点交给哪个工人去处理。**

维护器做的事情极其机械：
- 从优先队列中取出最高优先级的未处理节点
- 根据节点的类型和状态，决定分发给广度工人、深度工人还是攻击工人
- 接收工人的返回结果，更新节点状态
- 执行状态传播（如果所有子节点都验证通过，父节点也可以翻转为验证通过）
- 检测终止条件

维护器**不做任何语义判断**。它不理解节点里写了什么，它只看节点的类型标签和状态标签。这是整个系统可靠性的基础——控制流是确定性的，不依赖LLM的"想法"。

### 2. 广度引擎（Breadth Engine）——可能性的枚举器

广度引擎是一个LLM工人。它的任务是：**给定一个命题和一组约束，枚举出满足这些约束的多种可能的子方案或子命题。**

比如，给定"设计认知引擎的状态存储方案"和约束"必须支持多版本并行"，广度引擎可能输出三个候选方案：方案A用关系数据库、方案B用DAG、方案C用事件溯源。

广度引擎的关键特性：
- 它只看当前节点和传入的约束，不看全局树的其他部分
- 它的输出必须是结构化的（不是自由文本），每个候选方案都要附带核心假设和建议的客体类型标签
- 它不做评价，只做展开——"这里有这些选项"，不说"我推荐这个"

### 3. 深度引擎（Depth Engine）——命题的递归分解器

深度引擎也是一个LLM工人。它的任务是：**给定一个具体命题，把它分解成更小、更具体的子命题，直到子命题可以被外部工具直接验证，或者达到系统设定的递归深度上限。**

比如，给定"API密钥不能明文存储"，深度引擎可能分解为：
- 子命题1："API密钥必须从环境变量读取"
- 子命题2："日志模块必须配置密钥过滤规则"

深度引擎的关键特性：
- 它接收的输入是经过"精选"的逻辑切面——包含命题本身、相关的领域公理、局部定义、以及从根节点到当前节点路径上的所有已确立约束——但**不包含广度引擎的推导历史或其他工人的"对话记录"**
- 这种"上下文隔离"是刻意的：防止深度引擎被广度引擎的推理路径所引导，从而退化为对广度引擎输出的简单验证（分析判断），而不是独立的重新审视（综合判断）
- 当命题无法继续分解时，深度引擎尝试将其绑定到外部预言机（如代码检查工具、用户注入的公理）；如果绑定成功，标记为已验证；如果无法绑定也无法继续分解，标记为"假设"（Assumed）

### 4. 攻击引擎（Attack Engine）——反例的生成器

攻击引擎是第三类LLM工人。它的任务是：**给定一个局部子树，在有限的预算内，尽力找到能让这棵子树崩溃的反例。**

攻击引擎执行两种检查：
- **一致性检查**：子命题之间是否存在逻辑矛盾？（比如一个子命题要求单线程，另一个要求高并发）
- **覆盖完整性检查**：子命题是否覆盖了父命题的全部含义？（比如父命题是"系统安全性"，子命题只覆盖了"认证"和"授权"，遗漏了"数据加密"）

攻击引擎的关键特性：
- 它是**预算驱动的**，不是无限循环的。每次调用有固定的攻击次数（比如5次），从预设的攻击向量库中抽取不同视角（如并发竞争、单点故障、数据一致性）进行检查
- 它的输出必须是结构化的冲突报告，不是自由文本的评论。冲突报告必须指向具体的节点ID和被违反的约束ID
- 它不提供修改建议，只报告问题。修复工作由维护器分发给广度或深度引擎
- 它根据节点的客体类型使用不同的检查模式：对分析性客体（如代码逻辑），发现矛盾即阻断；对综合性客体（如架构哲学），发现不可调和的分歧则触发分叉而非阻断

### 5. 裁判引擎（Judge）——在本设计中被重新定义

在原始的辩论工具中，裁判是一个试图整合所有观点的LLM。在v4设计中，**裁判的角色被分解并嵌入到了维护器和攻击引擎中**：

- 维护器承担了"流程裁判"的角色：它决定什么时候一个节点可以被标记为已验证，什么时候需要回溯，什么时候触发分叉
- 攻击引擎承担了"质量裁判"的角色：它判断子树是否存在逻辑问题
- 不再有一个独立的"裁判"试图产出一个整合性的结论——因为这种整合正是导致平凡收敛的元凶

当攻击引擎发现一个节点处于"争议状态"（Contested）时，处理流程是：
1. 攻击引擎输出结构化的冲突报告
2. 维护器根据冲突类型和节点的客体类型决定处理方式
3. 如果是逻辑矛盾且客体是分析性的：标记为无效，回溯
4. 如果是不可调和的价值分歧且客体是综合性的：触发分叉，两个前提假设下分别继续展开
5. 如果是维度缺失：追加约束，触发广度引擎补充

### 6. 协议层（Protocol Layer）——系统的"宪法"

协议层不是一个独立的组件，而是贯穿整个系统的一套规则，确保LLM的输出可以被维护器机械地处理。

协议层的核心是一个**领域特定语言（DSL）**。系统强制要求LLM工人的输出必须符合DSL的语法，而不是自由的自然语言。DSL包含几种基本谓词：
- `CodeGrep(pattern)`：要求外部脚本在代码库中搜索特定模式
- `Metric(entity, operator, value)`：如 `Metric("恢复时间", "<", 50ms)`
- `AxiomRef(id)`：引用用户在启动时注入的领域公理
- `Conflict(node_id, violated_constraint_id)`：攻击引擎报告冲突

维护器内嵌一个DSL解析器。LLM的输出首先经过这个解析器：如果解析成功，结果被挂载到DAG节点上；如果解析失败（LLM输出了不合法的格式），系统重试，重试多次仍失败则将节点标记为"假设"并隔离。

协议层的意义在于：**它把LLM从一个"自由文本生成器"降级为一个"受限的DSL编译器前端"**。LLM的工作不是"思考"，而是把模糊的自然语言需求"编译"成精确的DSL表达式。编译成功，系统继续；编译失败，系统知道这里有问题。

## 双轨状态系统：构成性状态 vs 调节性分数

系统中每个节点有一个**状态标签**（构成性状态）和一组**元数据**（调节性信息）。

### 构成性状态（节点的硬状态）

每个节点有且仅有以下状态之一：
- **Unverified**：尚未处理
- **Verified**：已通过深度分解和攻击检查，且所有子节点也已验证
- **Assumed**：无法被外部预言机验证，也无法继续分解，作为假设悬置
- **Invalid**：被攻击引擎发现逻辑矛盾，已失效
- **Unsat**：不可满足——该节点及其依赖链存在根本性矛盾
- **Suspended**：因达到深度上限或预算耗尽而暂停

这些状态是**确定性的、可机械判断的**。维护器根据子节点的状态组合来计算父节点的状态，不需要LLM参与。

### 调节性信息（节点的软元数据）

每个节点还携带：
- **ProofToken**：记录该节点经过了多少轮攻击检查、深度引擎的调用记录、未解决的子问题列表。ProofToken不是"证明这个节点是对的"，而是"记录系统在这个节点上花了多少资源、做了什么检查"
- **约束列表**：攻击引擎发现的问题会被追加到这里，作为后续处理的输入
- **客体类型标签**（obj_type）：标记这个节点处理的是分析性问题还是综合性问题

### 直觉含义

构成性状态回答的是："这个节点现在处于什么阶段？"——这是系统控制流的依据。

调节性信息回答的是："系统对这个节点有多大的信心？这个信心是怎么来的？"——这是系统输出时向用户解释"为什么我认为这个结论可靠"的依据。

一个节点可以是 Verified 状态（所有子节点通过了检查），但它的 ProofToken 可能显示攻击预算只用了2次（意味着检查不够充分）。用户看到这个信息，可以决定是否要求系统增加预算重新检查。

## 终止条件的设计逻辑

系统有三种合法的终止方式：

### 1. 成功终止
根节点变为 Verified 或 Assumed。这意味着整棵命题树要么全部通过了验证，要么在某些地方做了明确标记的假设。系统输出完整的证明树、假设清单和分叉点列表。

### 2. 失败终止
根节点变为 Unsat。这意味着用户的初始需求在逻辑上存在根本性矛盾（比如同时要求CAP定理的三个属性）。系统输出从叶节点到根节点的冲突证明链，明确告诉用户"你的需求不可能同时满足，矛盾在这里"。

### 3. 预算耗尽终止
系统的总预算（LLM调用次数、递归深度上限）用完了，但仍有 Unverified 节点。系统输出当前状态快照，所有未完成的节点标记为 Suspended，供用户决定是否追加预算继续。

**关键设计原则：系统永远不会输出"问题已被完全解决"。** 即使根节点是 Verified，输出中也会包含 ProofToken（显示检查的深度和广度）和 Assumed 节点列表（显示哪些地方做了假设）。用户始终知道系统的结论是在什么条件和什么资源限制下得出的。

终止条件全部是可计算的：
- 深度上限：节点深度 ≥ MAX_DEPTH → Suspended
- 攻击预算：每个节点的攻击次数 ≤ k → 预算耗尽即停
- 全局预算：总LLM调用次数 ≤ N → 超出即停
- 状态传播：所有子节点状态确定 → 父节点状态可计算

没有任何终止条件依赖LLM的"我觉得够了"。

## Verified vs Hypothesis（Assumed）的区别和意义

这是系统输出中最重要的区分。

### Verified 节点
意味着：
1. 该命题已被深度引擎分解为更小的子命题
2. 所有子命题要么也是 Verified，要么已被外部预言机（代码检查工具、用户注入的公理）直接验证
3. 攻击引擎在给定预算内没有发现逻辑矛盾或覆盖缺失

**Verified 不等于"绝对正确"。** 它等于"在当前系统的检查能力和资源预算内，没有发现问题"。如果未来引入新的攻击向量或增加预算，原本 Verified 的节点可能被推翻。

### Assumed（Hypothesis）节点
意味着：
1. 该命题无法被外部预言机直接验证（不是代码事实，也不是用户注入的公理）
2. 深度引擎无法将其继续分解为可验证的子命题（或已达到深度上限）
3. 系统选择**假设它成立**，并在此假设上继续构建上层结论

**Assumed 节点是系统诚实承认自身局限性的标志。** 它告诉用户："我的结论依赖于这个假设。如果这个假设不成立，从这里往上的所有结论都需要重新审视。"

### 为什么这个区分重要

传统的LLM输出不区分"我确认的"和"我猜测的"——所有内容都以同样的自信语气呈现。v4 系统通过 Verified/Assumed 的显式标记，让用户能够：
- 快速定位结论中最薄弱的环节（Assumed 节点）
- 评估结论的可靠性（Verified 节点的比例和 ProofToken 的检查深度）
- 决定在哪里投入人工审查资源（优先审查 Assumed 节点）
- 理解不同结论分支的分歧根源（分叉点处的不同 Assumed 公理）

## 组件间的信息流总结

```
用户输入
    ↓
[Intake 解析] ← 唯一一次"元级别"LLM调用，确定根节点类型和领域公理
    ↓
[维护器] ← 纯确定性代码，维护DAG，调度工人
    ↓ 分发任务          ↑ 接收结果
    ↓                   ↑
[广度工人]  [深度工人]  [攻击工人]  ← 三类LLM工人，互不通信
    ↓                   ↑
[DSL 解析器] ← 协议层，验证工人输出的格式合法性
    ↓
[外部预言机] ← 代码检查工具、Linter、用户公理库等非LLM验证手段
    ↓
[维护器] ← 根据验证结果更新DAG节点状态
    ↓
[输出] ← 证明树 + 假设清单 + 分叉点 + 冲突证明链
```

工人之间**绝不直接通信**。所有信息都通过维护器中转，以DAG节点的形式存储和传递。这确保了：
- 没有"对话历史"的污染
- 每个工人看到的都是经过精选的、结构化的输入
- 系统状态在任何时刻都可以从DAG快照完整恢复

# 第二部分：完整示例走通

## 输入

- **问题**："设计一个能处理开放域大问题的认知引擎"
- **Objective**："产出一个尽量细节的可用设计方案及其辩护"

## 步骤 0：Intake 解析——维护器建立初始 DAG

系统启动时，执行唯一一次"元级别"LLM调用。这次调用极其受限，不负责生成方案，只负责分类和提取公理。

**Intake LLM 的输入**：
```
你的任务是将以下用户需求解析为结构化的任务描述。
用户需求："设计一个能处理开放域大问题的认知引擎，产出可用设计方案及其辩护"
请输出以下 JSON 格式，不要输出其他内容：
{
  "root_objective": "...",
  "obj_type": "Analytic | Synthetic | Mixed",
  "domain_axioms": ["...", "..."],
  "scope_constraints": ["...", "..."]
}
```

**Intake LLM 的输出**：
```json
{
  "root_objective": "设计一个认知引擎系统，能够处理开放域的复杂问题并产出结构化的设计方案",
  "obj_type": "Mixed",
  "domain_axioms": [
    "AX1: 系统必须能处理超过单次LLM上下文窗口的问题规模",
    "AX2: 系统的控制流必须可审计、可复现",
    "AX3: 系统必须能显式标记其结论的确定性程度"
  ],
  "scope_constraints": [
    "SC1: 系统以LLM作为核心计算单元",
    "SC2: 系统必须能在有限预算内终止"
  ]
}
```

**维护器的处理**：
维护器的DSL解析器验证这个JSON的格式合法性（字段是否齐全、obj_type是否属于枚举值）。验证通过后，维护器创建初始DAG：

```
DAG 初始状态：
Root Node (N0):
  id: hash(N0)
  content: "设计一个认知引擎系统，能够处理开放域的复杂问题并产出结构化的设计方案"
  status: Unverified
  node_type: Claim
  obj_type: Mixed
  constraints: [AX1, AX2, AX3, SC1, SC2]
  children: []
  proof_token: null
  depth: 0
```

优先队列中只有一个节点 N0。

## 步骤 1：广度引擎的第一次调用

维护器从优先队列取出 N0。N0 无子节点，需要展开。维护器分发给广度工人。

**广度工人的输入**：
```json
{
  "node_content": "设计一个认知引擎系统，能够处理开放域的复杂问题并产出结构化的设计方案",
  "constraints": ["AX1: 超过单次上下文窗口", "AX2: 控制流可审计", "AX3: 标记确定性程度", "SC1: LLM为核心计算单元", "SC2: 有限预算终止"],
  "parent_obj_type": "Mixed",
  "negated_options": [],
  "task": "枚举满足上述约束的顶层设计组件，每个组件附带核心假设和建议的 obj_type。输出为 JSON 数组。"
}
```

**广度工人的输出**（经DSL解析器验证后）：

## 步骤 3：攻击引擎发现一个 ConflictNode 的全过程

假设系统已经运行了若干轮，N4（攻击引擎的设计）的子树已经展开到一定深度。现在维护器调度攻击工人来检查 N4 的局部子树。

**N4 的局部子树**：
```
N4: 攻击引擎设计 (Unverified, Mixed)
  N4.1: 攻击引擎必须是预算驱动的，每次调用有固定次数上限 (Verified)
  N4.2: 攻击引擎必须检查子命题间的一致性 (Verified)
  N4.3: 攻击引擎的输出必须是结构化的 ConflictClause (Verified)
```

**攻击工人的输入**：
```json
{
  "dag_fragment": {
    "parent": {"id": "N4", "content": "攻击引擎设计", "obj_type": "Mixed"},
    "children": [
      {"id": "N4.1", "content": "预算驱动，固定次数上限", "status": "Verified"},
      {"id": "N4.2", "content": "检查子命题间一致性", "status": "Verified"},
      {"id": "N4.3", "content": "输出结构化 ConflictClause", "status": "Verified"}
    ]
  },
  "budget": {"max_queries": 3, "attack_vectors": ["Completeness", "Edge_Case", "Interaction"]},
  "obj_type": "Mixed"
}
```

**攻击工人的第一次探测**（攻击向量：Completeness / 覆盖完整性）：

攻击工人的提示词要求它检查：N4.1、N4.2、N4.3 是否覆盖了 N4（"攻击引擎设计"）的全部含义。

**攻击工人的输出**：
```json
{
  "conflict_found": true,
  "conflict_clause": {
    "conflict_type": "MissingCoverage",
    "target_node": "N4",
    "uncovered_aspect": {
      "description": "攻击引擎只检查子命题间的一致性，但不检查子命题对父命题的覆盖完整性",
      "symbolic_tags": ["Coverage_Completeness_Check"],
      "suggested_constraint": "Constraint::MustAddress('子命题对父命题的覆盖完整性检查')"
    }
  }
}
```

**维护器的处理**：

1. 维护器接收到 `MissingCoverage` 类型的冲突
2. 由于 N4 的 obj_type 是 Mixed，维护器不执行阻断（Invalid），而是执行**约束追加 + 广度扩展**
3. 将 `Constraint::MustAddress('覆盖完整性检查')` 追加到 N4 的约束列表
4. 触发广度工人重新展开 N4，要求在新的约束下补充缺失的子命题

**广度工人被重新调用后**，生成了新的子节点：
```
N4.4: 攻击引擎必须执行覆盖完整性检查——给定父命题P和子命题集合{Ci}，
      判断是否存在P的某个方面未被任何Ci覆盖 (Unverified)
```

N4.4 被挂载到 N4 下，加入优先队列继续处理。

**这就是 CEGAR 循环的一次完整迭代**：攻击引擎发现反例 → 维护器追加约束 → 广度/深度引擎在新约束下重新生成 → 攻击引擎再次检查。

## 步骤 4：Judge 处理一个 Contested 情况的流程

假设系统在处理 N5（协议层设计）时，深度引擎产出了两个互斥的子命题：

```
N5.3: 协议层应该要求LLM输出纯DSL，完全禁止自然语言
  (理由：消除一切语义歧义)
N5.4: 协议层应该允许LLM在DSL旁附带自然语言注释
  (理由：纯DSL会导致LLM输出质量严重下降，因为LLM的推理依赖自然语言的chain-of-thought)
```

攻击工人检查 N5 的子树时，发现 N5.3 和 N5.4 存在逻辑矛盾。

**攻击工人的输出**：
```json
{
  "conflict_found": true,
  "conflict_clause": {
    "conflict_type": "LogicalContradiction",
    "target_nodes": ["N5.3", "N5.4"],
    "description": "N5.3 要求完全禁止自然语言，N5.4 要求允许自然语言注释，两者互斥",
    "violated_constraint_id": null
  }
}
```

**维护器的处理**（这就是"Judge"的角色）：

1. 维护器检查 N5 的 obj_type：Mixed
2. 对于 Mixed 类型节点上的 LogicalContradiction，维护器检查冲突的性质：
   - N5.3 和 N5.4 的分歧不是逻辑错误（两者各自内部都是自洽的），而是**设计哲学的分歧**——对"DSL纯度 vs LLM输出质量"的不同权衡
3. 维护器判定这是一个**综合性分歧**（不是"2+2=5"那种分析性错误），触发 **State Fork**

**分叉操作**：
```
N5 被复制为两个并行节点：
N5-A: 协议层设计（前提：纯DSL，完全禁止自然语言）
  → 继承 N5 的所有已验证子节点
  → 删除 N5.4，保留 N5.3
  → 独立继续展开

N5-B: 协议层设计（前提：DSL + 自然语言注释混合模式）
  → 继承 N5 的所有已验证子节点
  → 删除 N5.3，保留 N5.4
  → 独立继续展开
```

分叉点被记录：
```json
{
  "fork_point": {
    "original_node": "N5",
    "condition_A": "纯DSL模式：LLM输出必须完全符合DSL语法",
    "condition_B": "混合模式：LLM输出DSL + 自然语言注释，注释不参与逻辑判断",
    "branch_A": "N5-A",
    "branch_B": "N5-B"
  }
}
```

两个分支在后续轮次中独立演化。最终输出会包含两套协议层设计方案，以及它们的分歧根源。

## 输出解读

这个输出告诉用户：

1. **系统产出了两套设计方案**（Obj_A 和 Obj_B），它们共享大部分架构（维护器、广度引擎、深度引擎、攻击引擎的设计），但在协议层的设计上存在根本性分歧。

2. **分歧的根源被精确定位**：是否允许自然语言注释。用户可以根据自己的优先级（DSL纯度 vs LLM输出质量）选择方案，或者先验证关键假设（N5-A.3.1）再做决定。

3. **每套方案的薄弱环节被明确标记**：Obj_A 有一个 CRITICAL 级别的假设（纯DSL不会严重降低LLM输出质量），Obj_B 有一个 MEDIUM 级别的假设（自然语言注释不会被误解析）。用户知道在哪里投入审查资源。

4. **系统承认了自身的局限性**：两个全局 Issue 说明了当前设计中尚未完全解决的根本问题（分解完整性和分叉判定的可靠性）。

5. **系统可以续跑**：workbench_snapshot 包含完整的DAG状态，用户可以追加预算继续处理 Suspended 节点，或者修改公理后重新运行。

这就是 v4 认知引擎的一次完整运行：从模糊的自然语言需求出发，通过结构化的分解、攻击和分叉，产出多套带有明确假设标记和分歧定位的设计方案，而不是一篇看似完整实则模糊的"最终结论"。
