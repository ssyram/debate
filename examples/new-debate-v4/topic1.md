# v4 认知引擎：整体系统设计的自由探索

## 背景：设计动机与核心想法

> 原始设计文本：抽象来说，我们有广度引擎，负责：根据需求（第一轮） / 根据反馈（后面轮次），其实根据反馈也是根据需求，我们应该有个全貌综合器负责高精度维护全方案和全貌，然后通过广度引擎产生张力，中间通过一个分解引擎将 claim 和设计爬出来，作为一个个问题扔给深度引擎，深度引擎负责稳固，也就是逐个向前拆解其理由/原因，再根据其直觉性程度再迭代拆，拆到一个地方直觉程度足够高，我们可以将其作为原子公理的时候就保留下来（这些东西未来还要面对冲突检查，可能好几个东西之间是有冲突的，要么找到调和，从而进一步确定系统范围和/或打磨词义精确描述，要么产生分裂和分别的讨论，即不同情况下采取不同体系），精度引擎/清晰度引擎在深度引擎和全貌综合器中需要实时进行更高的掌控和检查，它专门负责找漏洞执行（逻辑）攻击，攻击纳入 worker 考虑范围，进一步思考和改进整个过程。如果深度引擎否定了方案，则需要回馈到综合器，综合器简单确信判断哪些地方存在问题，如果判断本身做不了，扔给深度引擎负责命题性确认，如果判断是可马上简单修复（替换原来已经有的合适部件方案，直接塞进去）就直接修复不用管，如果是发现有空缺 —— 导致某一整块都用不了了，直接反馈给广度引擎，重新根据需求生成。也就是维护器负责从头到尾维护我们需要的总输出的（一个或者多个的）全貌，维护器会发出描述性请求和命题性请求，分别发给广度引擎（较低精度素材引擎）和深度引擎（高精度命题证明/分解引擎），这中间精度引擎和清晰度引擎分别爬过维护器和深度引擎的部分，执行攻击检查，深度引擎负责命题递归分解，精度引擎则负责攻击这些分解步骤里面的论证性，强迫深度引擎做出回答，维护器执行维护的时候精度引擎则负责逐个部件敲打攻击，生成辅助性宏观命题，负责在维护器执行的每一步都提问，强迫维护器做深思 —— 但是维护器深思只有确信和不确信两种结果，不确信马上找深度引擎处理，确信则可以继续执行。这个方案如何？

这个系统的起点是对上面原始文本的反思。原始文本的根本问题不是能力不够，而是设计上有三个结构性缺陷：角色之间职责重叠（多个组件都在做"检查"和"决策"），控制流存在潜在死锁（攻击引擎和深度引擎之间可以形成无终止的互相触发循环），以及状态分散在多个组件中（没有单一权威来源）。核心洞察是：必须有一个权威的全局状态拥有者，而攻击/精度/清晰度能力不应该是平级引擎，而是一套横切策略。由此产生了 3+1 架构：中央维护器（唯一全局状态拥有者）+ 广度引擎（设计空间展开器，处理描述性请求）+ 深度引擎（递归分解检查器，处理命题性请求）+ 横切攻击策略（不是平级引擎，而是挂在各阶段的检查能力）。各引擎可以多 workers 并行处理，相互之间互不了解。

在这个基础上，用户对维护器的定位有一个新的核心想法：维护器应该是"铸造团队"而不是"掌控管理器"。这个区分的意义在于弱化维护器的决策能力——迫使它只在高度确信时才自主行动，凡是不确信的事项必须借助双引擎。维护器的工作方式像蜘蛛爬网：用脚手架从宏观到微观逐步设计，逐个零件拆解，然后逐个将对应话题分发给两个引擎处理。维护器维护的对象可能是庞大的"超级巨物"——完整描述超过 10M token 的项目，因此维护器不可能在单次上下文中持有全貌，必须在宏观和微观之间分层维护。维护器内部可以同时维护一个或多个体系，多个体系在回复用户质询时可能都有价值。

整个系统的顶层目标是：给定一个通用的、可以很庞大的问题（例如"评判 Linux 代码框架并做代码 Review"、"设计一个新时代嵌入式系统体系"、"德川家为什么能取得丰臣天下"），给定对输出 Objective 的定义，系统实际产出：一个或多个 Objectives（可配置）；每个 Objective 的详细合理性论证和支撑（证明树），连带单 Objective 内部的讨论证明树；Objectives 之间的区分条件，即采用不同公理的核心分岔点；未决条件和对应 Objective 的 reference；以及中央维护器的中间体系状态本身，用以允许用户质询和续跑。用这个系统设计这个系统本身是首要测试用例。用户目前没有固定的设计方案，非常 flexible，整个设计空间是开放的。
