# COURT WORKING MODE：规则合理性、卡顿与 Bug 分析 裁判总结

> 2026-03-09T16:57:55.406750

# 最终裁定

## 第一部分：白话版结论

### 规则 1：永远 run_in_background: true，无一例外

**裁定：需修正**

康德在第三轮指出了一个真实的边界问题：如果用户只是问"你理解我的意思吗？"，主线程为了遵守"无一例外"，也必须创建一个后台Task去"调查用户意图"，然后自己停下等回报。这显然荒谬——一句直接的澄清回复被强行异步化，徒增一轮延迟。

**具体例子**：用户说"我说的重构是指提取函数，不是重写模块，你明白吗？"主线程本可以直接回复"明白了，我调整计划"，但规则1要求任何动作都必须后台化，导致主线程反而要派一个Sub-Agent去"确认用户意图"，然后等回报，再回复。

**一句话总结**：规则1应限定为"凡Task工具调用一律background"，但主线程的直接协调性响应（澄清、确认、计划调整）不受此约束。

---

### 规则 2：绝对不自己做任务

**裁定：需修正**

三方辩手在最后一轮达成了实质共识：规则2的"绝对不做"在边界处存在类型越界问题。Ssyram精确指出——当Sub-Agent返回`blocker_type: "missing_dep"`时，主线程要据此决定"派谁去装依赖"，这个判断算不算"自己做任务"？如果算，主线程就瘫痪了；如果不算，"绝对"二字就是空话。

**具体例子**：Sub-Agent报告"auth.py第220行有依赖冲突，无法继续"。主线程需要读懂这句话、判断应该派一个"安装依赖"的任务还是"换实现方案"的任务。这个判断涉及对blocker的理解，但不涉及亲手改代码。当前规则2没有区分这两者。

观察者提供的派发示例A更进一步暴露了问题的另一面：主线程写出了完整的修复伪代码（`write_text`替换为临时文件、`add()`增加`flush=True`），实质上已经完成了设计工作，Sub-Agent只是打字员。这是规则2被实质掏空的反面——不是"做太少"，而是"做太多但假装没做"。

**一句话总结**：规则2应改为"主线程不得直接执行领域对象变更（读写业务文件、运行业务代码），但可执行协调性判断（分析blocker类型、决定下一步派发策略）"。

---

### 规则 3：使用工具完成所有任务

**裁定：合理，无需修正**

辩论中无人对规则3提出实质性攻击。它是规则2的执行层保障——既然不自己做，就必须通过工具做。三方的争论集中在"什么算任务"的边界上，而非"是否应该用工具"。

**具体例子**：当主线程需要知道某个文件的内容时，它必须通过Task工具派Sub-Agent去读，而不是自己凭记忆猜测。这条规则本身没有歧义。

**一句话总结**：规则3本身清晰可执行，其边界问题已被规则2的修正所覆盖。

---

### 规则 4：主线程只做统筹协调

**裁定：合理，但需与规则2的修正联动**

规则4定义了主线程的角色——统筹协调者。Ssyram提出的"控制面与数据面隔离"框架实际上就是规则4的形式化表述。问题不在规则4本身，而在它与规则2的边界未对齐。

**具体例子**：主线程维护TODO列表、决定任务优先级、根据回报调整计划——这些都是规则4允许的。但当主线程为了"协调"而开始写具体的代码修复方案时（如派发示例A），它就越过了规则4的边界，进入了规则2禁止的领域。

**一句话总结**：规则4本身合理，但必须与修正后的规则2形成清晰的权责边界——协调性判断属于规则4，领域对象变更属于规则2禁区。

---

### 规则 5：每次派完人立即停下

**裁定：有Bug，需修正**

这是辩论中争议最激烈的规则。Linus构造了一个无法反驳的失活路径：

```
t0 TODO=open
t1 spawn A(background)
t2 按R5"立即停下"
t3 A卡死且无回报
t4 因无timer/回调，主线程永不恢复
```

康德试图将此降格为"不完备而非Bug"，但Linus的反驳有力：规则系统声称"无论任何时候都可流畅执行"，只要规范允许这条失活路径存在，就不合格。这不是实现者粗心，而是规则文本本身没有定义恢复触发器。

Ssyram从另一个角度加强了攻击：在LLM的响应式运行环境中，主线程被挂起后，如果Sub-Agent崩溃或无限循环，根本没有OS级定时器来唤醒主线程。所谓的"超时事件"是一个幽灵事件——它在规则中被假设存在，但在实际平台中无从产生。

**具体例子**：主线程派了一个Sub-Agent去重构一个大文件，然后按规则5停下。Sub-Agent因为token超限而静默失败，永远不返回结果。主线程就这样永远挂着，TODO永远是open状态，但没有任何机制能让它醒过来。

**一句话总结**：规则5应改为"派发后结束本轮输出，但结束前必须完成：记录TODO状态、登记task_id、声明下一唤醒条件（回报/用户消息/平台超时）"。

---

### 规则 6：不要一次派太多人

**裁定：合理，无需修正**

辩论中规则6被提及但未受到实质攻击。它是一条实用的资源管理约束，防止主线程同时派出过多Sub-Agent导致上下文爆炸或回报冲突。

**具体例子**：用户要求修改5个文件。主线程不应该一次派5个Sub-Agent同时改，而应该按依赖关系分批派发，每批2-3个。这条规则的合理性在辩论中未被质疑。

**一句话总结**：规则6是合理的工程约束，当前表述足够清晰。

---

### 规则 7：不清晰先询问/计划

**裁定：需修正（与规则9存在竞态条件）**

Linus在第一轮就精确指出了R7与R9的竞态条件：

```
t0 用户任务不清晰
t1 主线程按R7询问用户
t2 同时为了"不彻底停下"按R9派了一个explore agent
t3 用户随后回复，改变任务边界
t4 explore agent也回报了旧边界下的结果
```

此时旧agent的回报是过期的，但规则集中没有版本控制或取消机制来处理这种情况。Ssyram后来提出的Epoch状态机方案（给每次派发打上逻辑时钟版本号，过期回报静默丢弃）是正确的修正方向。

**具体例子**：用户说"帮我优化性能"，主线程不确定是优化哪部分，于是一边问用户"你指的是数据库查询还是前端渲染？"，一边派了个agent去做性能分析。用户回复"我说的是数据库"，但agent已经在分析前端渲染了。这个过期结果如果被写入TODO，就会污染后续决策。

**一句话总结**：规则7应附加"当用户输入导致任务边界变更时，主线程必须能标记/丢弃基于旧边界的过期回报"。

---

### 规则 8：精确TODO维护

**裁定：合理，但依赖规则10的修正**

规则8本身的要求——维护精确的TODO列表——是正确的。问题在于它的执行质量完全取决于Sub-Agent回报的信息质量。如果规则10允许Sub-Agent在partial/blocked时只返回一句"没做完"，主线程就无法精确更新TODO。

Linus的论证最为直接：没有`blocker_type`，主线程无法区分"权限不足"和"上下文超限"；没有`completed_steps`，重派时只能重复已做步骤。

**具体例子**：Sub-Agent报告"auth.py修改未完成"。主线程更新TODO为"auth.py: 未完成"。然后呢？它不知道是哪一步卡住了，不知道已经改了什么，不知道该派什么样的后续任务。TODO虽然"更新"了，但信息量为零。

**一句话总结**：规则8本身合理，但其可执行性依赖于规则10提供结构化的partial/blocked回报。

---

### 规则 9：TODO未完成时不能彻底停下

**裁定：需修正**

康德对规则9的分析最为精确：它本质上是一个safety约束（"不可彻底放空"），但被错误地期望承担liveness保证（"必然继续推进"）。Linus用一个同义反复揭示了这一点：

```
if todo_open and bg_exists and bg_progress=false_forever:
    system = stuck
```

"存在后台任务"不等于"后台任务会推进"。规则9只检查`bg_exists`，不检查`bg_progress`。

**具体例子**：主线程派了一个Sub-Agent，TODO显示"进行中"，规则9满足（有后台任务在跑）。但这个Sub-Agent实际上已经陷入了无限循环，永远不会返回。系统表面上"没有彻底停下"，实际上已经死了。

**一句话总结**：规则9应改为"TODO未完成时，系统必须处于以下状态之一：有可推进的后台任务、等待用户澄清、已注册恢复动作"——重点是"可推进"而非仅仅"存在"。

---

### 规则 10：信任Sub-Agent，最少足量信息

**裁定：有Bug，需重写（派发端和回报端均有问题）**

这是本次辩论中被攻击最彻底的规则，且三方辩手加上观察者形成了罕见的四方共识：规则10在两个方向上都有Bug。

**回报端Bug**（前几轮已确认）：partial/blocked时"极简回报"导致主线程无法做出有效决策。三方一致同意需要结构化失败协议。

**派发端Bug**（观察者本轮揭示）：主线程在派发时过度描述，实质上替Sub-Agent完成了认知工作，违反了规则2的精神。三个派发示例清晰展示了这一模式：
- 示例A：主线程写出了完整的修复伪代码，Sub-Agent退化为打字员
- 示例B：主线程把要写入的全部内容都写进了prompt，Sub-Agent只做文件I/O
- 示例C：主线程把大段原文嵌入派发，导致prompt本身膨胀

Linus给出了最清晰的边界划分——必须给的只有三类：目标（做什么）、作用域（在哪做）、约束（什么算完成）。不该给的：实现细节、内容总结、冗余的操作指令。

**各辩手对观察者意见的吸收情况**：

- **Linus**：完全吸收。明确指出"主线程提前做了本该由Sub-Agent完成的认知工作"是核心违规模式，给出了三类必须给/不该给的精确边界，提出了R11（可由Sub-Agent自行获取的信息不得重复内嵌），并给出了三个合规重写示例。重写质量最高——简洁、目标明确、不越权。

- **Ssyram**：完全吸收。从形式化角度将问题命名为"符号执行泄漏"和"影子实现"，给出了"必须给vs禁止给"的类型化边界，提出了量化约束（不超过5行代码/100字非结构化指令），并给出了合规重写示例。但其量化约束（5行/100字）过于刚性，在复杂任务中可能不切实际。

- **康德**：完全吸收。从义务论角度区分了"必要约束"与"替Sub-Agent思考"，提出了与Linus高度一致的R10'和R11，并给出了合规重写示例。其示例C的处理最为巧妙——"消息内容以用户原话为准，不在此重复展开"。

**具体例子**（派发端）：主线程收到用户请求"修复runner.py的bug"。违规做法：主线程自己分析代码、写出修复方案、告诉Sub-Agent"把第440行改成这样"。合规做法：主线程告诉Sub-Agent"修复runner.py中的日志相关问题，涉及run()和resume()路径，自行定位并修复，完成后汇报改动点"。

**具体例子**（回报端）：Sub-Agent修改auth.py只完成了一半。违规回报："没做完。"合规回报：`{status: "partial", touched_files: ["auth.py"], completed_steps: ["extract helper"], blocked_step: "merge session", blocker_type: "context_limit"}`。

**一句话总结**：规则10必须重写为双向契约——派发端只给目标/作用域/约束，禁止预写实现方案；回报端success可极简，partial/blocked必须结构化。

## 第二部分：规则集修正摘要

### 最终推荐的COURT WORKING MODE规则集

**规则 1'**：凡Task工具调用一律使用`run_in_background: true`。主线程的直接协调性响应（对用户的澄清、确认、计划说明）不属于Task调用，不受此约束。

**规则 2'**：主线程不得直接执行领域对象变更（修改业务文件、运行业务代码、生成实现方案）。主线程可执行协调性判断：分析结构化回报中的blocker类型、决定派发策略、调整TODO优先级。判断标准：如果该动作的输出是"改了什么文件/写了什么代码"，则属于领域变更，禁止；如果输出是"接下来派谁做什么"，则属于协调，允许。

**规则 3**：（保持原文）使用工具完成所有任务。

**规则 4**：（保持原文）主线程只做统筹协调。与规则2'联动：协调性判断归规则4管辖，领域变更归规则2'禁区。

**规则 5'**：每次派发后结束本轮输出。但结束前必须完成以下最小义务集：(a) 更新TODO状态；(b) 登记已派发的task_id；(c) 声明下一唤醒条件（Sub-Agent回报 / 用户消息 / 平台超时事件之一）。若无法声明任何唤醒条件，则不得进入挂起态，必须先向用户报告当前状态。

**规则 6**：（保持原文）不要一次派太多人。

**规则 7'**：不清晰先询问/计划。附加：当用户输入导致任务边界变更时，主线程必须递增逻辑版本号（Epoch），基于旧Epoch派发的任务回报在到达时应被标记为过期，不得用于更新当前TODO。若底层平台支持TaskCancel，应显式取消过期任务。

**规则 8**：（保持原文）精确TODO维护。其执行质量由规则10'的结构化回报保障。

**规则 9'**：TODO未完成时，系统必须处于以下状态之一：(a) 存在可推进的后台任务（非仅"存在"，而是"未超时且未报告blocked"）；(b) 等待用户澄清；(c) 本轮已注册恢复动作（如补派scout agent）。若三者均不满足，主线程必须立即向用户报告阻塞状态并请求指令。

**规则 10'**（双向信息契约）：
- **派发端**：仅提供目标（做什么）、作用域（在哪做）、约束（什么算完成、什么不能做）、必要的上下文定位符（文件路径、相关函数名）。不得预写实现方案、不得嵌入Sub-Agent可通过一次读取自行获得的材料、不得包含冗余的操作指令（如"先读取文件"）。例外：用户明确指定的精确补丁或必须原样传递的用户消息内容。
- **回报端**：success可极简（完成摘要即可）。partial/blocked必须返回结构化报告，最小字段集：`{status, touched_files, completed_steps, blocked_step, blocker_type, next_recommended_action}`。

**规则 11**（新增·派发冗余禁令）：凡可由Sub-Agent通过读取指定文件、运行指定命令、或查阅已有上下文自行获得的信息，主线程不得在派发prompt中重复展开。判断标准：如果删除该段信息后Sub-Agent仍能通过一次工具调用获取等价内容，则该信息为冗余，应删除。

### 实现难度最高的3个约束及其风险

**1. 规则 2'的"协调性判断 vs 领域变更"边界**
- **难度**：这是一个连续光谱而非二元分类。"分析blocker并决定下一步"在简单场景下是纯协调（blocker_type=missing_dep → 派安装任务），但在复杂场景下可能需要理解业务逻辑才能做出正确判断（blocker_type=design_conflict → 需要理解架构才能决定方案）。
- **风险**：主线程可能以"协调"为名行"实现"之实（如派发示例A中的"影子实现"），或者过度保守地拒绝一切判断导致瘫痪。
- **缓解**：以输出类型为判断标准——输出是"派谁做什么"则合规，输出是"怎么改代码"则违规。

**2. 规则 7'的Epoch版本控制**
- **难度**：当前LLM Agent平台（包括Claude Code）没有原生的逻辑时钟或TaskCancel机制。Epoch需要在prompt层面模拟，依赖主线程自律维护版本号，且过期回报的"静默丢弃"在实际执行中可能导致信息损失（过期回报中可能包含仍有价值的部分发现）。
- **风险**：版本号维护增加主线程认知负担；在快速迭代场景中Epoch频繁递增可能导致大量有效工作被丢弃。
- **缓解**：过期回报不必完全丢弃，可标记为"参考信息，不得直接更新TODO"，由主线程在新Epoch下判断是否仍有价值。

**3. 规则 10'回报端的结构化失败协议**
- **难度**：Sub-Agent是LLM，不是确定性程序。要求它在partial/blocked时严格返回指定JSON字段集，需要在派发时就嵌入回报格式要求，这本身又增加了派发端的信息量，与派发端"极简"原则存在张力。
- **风险**：Sub-Agent可能返回不完整或格式错误的结构化报告；`blocker_type`的分类体系（context_limit / test_fail / dependency_unknown / permission_denied等）需要预定义，否则Sub-Agent会自由发挥导致主线程无法程序化处理。
- **缓解**：将回报格式要求写入系统级prompt（而非每次派发时重复），并预定义有限的blocker_type枚举。

### 矛盾消解

本次辩论中发现的核心矛盾及其最小修改：

**矛盾1：R5"立即停下" vs R9"不能彻底停下"**
- 消解：R5'将"停下"重定义为"结束输出但必须先完成最小义务集"，R9'将"不能停下"重定义为"必须处于三种可推进状态之一"。两者不再冲突——R5'规定停下前的义务，R9'规定停下后的状态约束。

**矛盾2：R2"绝对不做" vs R4"统筹协调"需要理解回报**
- 消解：R2'引入"领域变更 vs 协调性判断"的区分，以输出类型为判断标准。这是最小修改——只加了一个判断准则，没有改变规则的基本方向。

**矛盾3：R10"极简信息" vs R8"精确TODO"需要充分信息**
