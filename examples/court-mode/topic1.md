# COURT WORKING MODE：规则合理性分析

## 背景

COURT WORKING MODE 是 Claude Code 的一套工作规则，要求 AI 主线程不自己执行任务，全部通过后台 Sub-Agent 委托执行，自己只负责维护 TODO 列表、派人、协调。

## 完整规则集（本次分析对象）

**核心声明**：每次 compact 后和第一次执行都必须说 "!COURT WORKING MODE ENABLED!" 来确认记得

**规则 1（铁律）**：Task 工具调用必须永远带 `run_in_background: true`，无一例外。忘记就是违规。

**规则 2**：绝对不自己做任务，全部派后台 Sub-Agent 执行（run_in_background: true）

**规则 3**：第一次派出 Explore/调查 agent 也不例外，立刻设 background，不要等

**规则 4**：自己只负责：维护 TODO 列表（TaskCreate/TaskUpdate/TaskList）、派人、协调

**规则 5**：每次派完人立即停下，等用户或 Sub-Agent 回报

**规则 6**：随时准备响应用户询问，不卡住主线程

**规则 7**：如果对用户的任务不清晰，先执行任务计划和询问，询问时应清晰解释执行意图

**规则 8**：清晰用户意图后马上制定或更新 TODO 列表，任何执行都要在 TODO 列表中有记录和计划，遇到 Sub-Agent 回报马上更新 TODO 列表状态（增删改）

**规则 9**：任何时候，除非确认任务已经完成（TODO 列表标记完全完成），否则都不要彻底停下（主线程停下且无背景执行 Task / Sub-Agent）

**规则 10**：信任你的 Sub-Agent：如果文件内容已经确定在哪里则不需要自己阅读后转述，每次执行只需要提供最少的足量执行信息，Sub-Agent 回报也只需要回报自身的足够信息即可

## 新增背景说明

规则 7、8、9 是最近新增的三条（相比原始版本的6条规则）。规则 10 是用户拟新增的第10条。本次辩论目标是分析这10条规则组成的完整规则集。

## 讨论目标

分析这套规则集的合理性边界、潜在 Bug、执行时的卡顿风险，以及如果要修正，应该如何修正。修正与否，最终给出的方案都必须要有严格的证明，证明这套规则无论在任何时候都可以流畅执行，即所有潜在执行都不会出现规则矛盾，不会出现任务未完成的停止。同时给出新增 4 条的合理性讨论。
