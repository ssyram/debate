# COURT WORKING MODE：规则合理性、卡顿与 Bug 分析 裁判总结

> 2026-03-09T15:23:22.082917

# 最终裁定

## 第一部分：白话版结论

### 规则 1：永远 run_in_background: true，无一例外

**裁定：需修正**

康德在第三轮指出了一个真实的边界问题：如果用户只是问"你理解我的意思吗？"，主线程为了遵守"无一例外"，也必须创建一个后台 Task 去"调查用户意图"，然后自己停下等回报。这显然荒谬——一句直接的澄清回复被强行异步化，徒增一轮延迟。

**具体例子**：用户说"我说的重构是指提取函数，不是重写模块，你明白吗？"主线程本可以直接回复"明白了，我调整计划"，但规则 1 要求任何动作都必须后台化，导致主线程反而要派一个 Sub-Agent 去"确认用户意图"，然后等回报，再回复。这是把简单的同步确认变成了不必要的异步链路。

**一句话总结**：凡 Task 调用一律 background，但主线程的直接协调性响应（澄清、确认、简短回答）不受此约束。

---

### 规则 2：绝对不自己做，只管协调

**裁定：需修正**

这是辩论中暴露最深层矛盾的规则之一。Ssyram 在质询康德时提出了一个致命问题：当 Sub-Agent 返回 `{status: "partial", blocker_type: "missing_dep"}` 时，主线程要决定下一步该派谁去做什么——这个"分析 blocker 并生成 next task"的动作，到底算"协调"还是"自己做任务"？

如果算"自己做任务"，那主线程在任何 partial 回报面前都瘫痪了，因为它连"理解失败原因然后决定补救"都不能做；如果算"协调"，那"绝对不自己做"的"绝对"就需要收窄。

**具体例子**：Sub-Agent A 回报"auth.py 重构失败，因为依赖 jwt 库版本不兼容"。主线程需要读懂"jwt 库版本不兼容"意味着什么，才能决定是派 B 去升级依赖还是派 C 去用旧 API 重写。这个判断涉及领域理解，但又确实是协调决策。规则 2 的"绝对"一词把这种必要的协调性判断也禁掉了。

**一句话总结**：主线程不得直接执行领域对象变更（读写文件、运行测试），但可以执行基于 Sub-Agent 回报的协调性判断（分析阻塞原因、决定补派策略）。

---

### 规则 3：严格的工具使用纪律

**裁定：合理，无需修正**

辩论中无人对规则 3 发起实质性攻击。它要求主线程通过工具调用（而非自己编造输出）来完成任务，这是 COURT 模式的基础架构假设，与其他规则无冲突。

**具体例子**：主线程需要知道某个文件的内容时，必须通过 `Read` 工具或派发 Sub-Agent 去读取，而不能自己凭记忆编造文件内容。这是对幻觉的结构性防御。

**一句话总结**：工具纪律是整个模式的地基，当前表述清晰可执行。

---

### 规则 4：主线程统筹，不做具体活

**裁定：需与规则 2 联动修正**

规则 4 本身的方向是对的：主线程是调度器，不是执行器。但正如规则 2 的分析所示，"统筹"与"做具体活"之间的边界在 partial/blocked 场景下不清晰。Ssyram 的质询"next_task 的类型签名属于谁"直接暴露了这一点。

**具体例子**：Sub-Agent 报告"测试有 3 个失败"。主线程要把这 3 个失败分配给不同的 Sub-Agent 去修——这个"拆分失败项并分配"的动作，是"统筹"还是"做具体活"？若 Sub-Agent 的回报里没有结构化的失败列表，主线程就得自己解析测试输出来拆分任务，这就越过了规则 2/4 的边界。

**一句话总结**：规则 4 的方向正确，但需要与规则 2 联动明确"协调性判断"属于统筹职责范围。

---

### 规则 5：每次派完人立即停下

**裁定：有 Bug，必须修正**

这是辩论中争议最激烈的规则。三位辩手最终形成了一个关键共识：**"立即停下"没有定义停下前的最小义务集，也没有定义恢复触发器。**

Linus 的核心攻击路径始终成立：
```
t0 建 TODO → t1 派 A(background) → t2 按 R5 停下 → t3 A 卡死 → t4 无恢复入口
```

康德试图将此降格为"文本不完备而非规则矛盾"，但在第三轮自己也承认了：**"有 Bug，不是因存在坏实现，而是因'立即停下'未指明停下前的最小义务集。"**

Ssyram 从平台约束角度做了一个重要澄清：在当前 Claude Code 架构下，单次 Tool Call 后确实必须 yield，所以"立即停下"部分反映了平台现实。但即便如此，停下前仍需完成必要的状态登记。

**具体例子**：用户要求分析 A、B、C 三个模块。主线程派出 Agent 分析 A，然后"立即停下"。但它没有记录"B、C 待派"，也没有声明"等 A 回报后继续"。结果 A 卡死后，主线程不知道自己还有 B、C 要做，也不知道该在什么条件下恢复。系统进入"假死"状态——TODO 未完成，但没有任何东西会触发下一步。

**一句话总结**：派发后可以结束本轮输出，但结束前必须完成三件事：记录 TODO 状态、登记 task 标识、声明下一唤醒条件（回报/用户消息/超时）。

---

### 规则 6：随时准备响应用户，不卡住主线程

**裁定：合理，但依赖规则 5 的修正**

规则 6 本身的方向没有问题。辩论中它主要作为规则 5/9 冲突的"第三方受害者"出现——如果规则 5 让主线程进入不可恢复的等待，规则 6 的"随时响应"就成了空话。但一旦规则 5 被修正为包含明确的恢复机制，规则 6 就可以自洽地执行。

**具体例子**：后台 Agent 正在分析模块 A，用户突然问"现在进度如何？"主线程应该能够立即回答"A 正在分析中，B/C 待派"，而不是因为"停下等回报"而无法响应。修正后的规则 5 保证了主线程有足够的状态信息来回答这类问题。

**一句话总结**：规则 6 本身合理，前提是规则 5 修正后主线程确实保持了可响应状态。

---

### 规则 7：不清晰先问再做

**裁定：合理，但与规则 9 存在竞态条件需要处理**

Linus 在交叉质询中提出了一个精确的竞态场景：
```
t0 用户任务不清晰
t1 主线程按 R7 询问用户
t2 同时按 R9 派了 explore agent（为了"不彻底停下"）
t3 用户回复，改变任务边界
t4 explore agent 回报了旧边界下的结果
```

这个场景下，旧 agent 的结果已经过时，但没有机制标记它为无效。Ssyram 后来提出了 Epoch（逻辑时钟）方案来解决这个问题，这个方向是对的。

**具体例子**：用户说"帮我重构登录模块"，主线程觉得不清晰，问"你指的是 OAuth 部分还是整个 auth 流程？"同时为了不闲着，派了个 Agent 去探索登录模块结构。用户回复"其实我改主意了，帮我重构支付模块"。但探索 Agent 还在跑，回来的结果是关于登录模块的，如果主线程不加区分地接受这个结果更新 TODO，就会把过时信息混入新计划。

**一句话总结**：规则 7 本身合理，但需要配合版本化机制（Epoch），确保用户澄清导致任务边界变化时，旧的后台结果能被正确丢弃。

---

### 规则 8：精确维护 TODO 列表

**裁定：合理，但被 Ssyram 过度解读为"活锁驱动器"**

Ssyram 试图论证规则 8 会驱动活锁——"为了满足规则 8（必须更新 TODO 状态），主线程只能继续派发 Scout C"。Linus 在交叉质询中精确反驳了这一点：**规则 8 只要求"状态记录"，不要求"重复派发"。** 主线程完全可以将 TODO 标为 `blocked(env_limit)` 然后转向用户询问，这同样满足规则 8。

**具体例子**：Agent A 回报"auth.py 太大，只处理了前 300 行"。主线程按规则 8 更新 TODO：`[blocked] auth.py 重构 — 环境限制，已完成 1-300 行，301+ 待处理`。然后可以选择派分块 Agent、问用户是否缩小范围、或标记为需要人工介入。规则 8 不强制任何一种选择。

**一句话总结**：规则 8 是记录纪律，不是调度命令；它不驱动活锁，但依赖规则 10 的修正来获得足够的更新信息。

---

### 规则 9：未完成不彻底停下

**裁定：需修正**

三位辩手在规则 9 上形成了一个渐进共识：
- Linus 最初说它与规则 5"直接冲突"，后来修正为"规则 9 的活性保证不充分"；
- 康德始终认为它是"必要但不充分条件"，最终同意需要修正；
- Ssyram 指出 `BgRunning ≠ ∅` 不等于"系统在推进"。

核心问题是：规则 9 只检查"是否存在后台任务"，不检查"后台任务是否在推进"。一个永久挂起的 Agent 满足规则 9 的字面要求，但系统实际已经停滞。

**具体例子**：TODO 有 5 项未完成，后台有 1 个 Agent 在跑但已经 10 分钟没有任何输出。规则 9 说"不要彻底停下"——好，没彻底停下，有个 Agent 在后台呢。但系统实际上在等一个可能永远不会回来的 Agent。用户体验就是"卡住了"。

**一句话总结**：修正为"TODO 未完成时，系统必须处于以下状态之一：有可推进的后台任务在运行、正在等待用户澄清、已安排恢复动作（超时重派等）"。

---

### 规则 10：极简信息流，信任 Sub-Agent

**裁定：有 Bug，必须修正**

这是辩论中共识度最高的结论。三位辩手从不同角度证明了同一个问题：**"极简回报"在成功场景下合理，在失败/部分完成场景下致命。**

- Linus 要求结构化失败协议（JSON Schema）；
- Ssyram 从 CEGAR 角度论证"极简回报剥夺了主线程精化所需的冲突子句"；
- 康德最终也同意"'足够'未类型化"是核心问题。

**具体例子**：Agent A 被派去重构 auth.py，失败了，按规则 10 只回报"任务未完成"。主线程现在要决定下一步：是重试？换策略？拆分任务？缩小范围？它什么都决定不了，因为它不知道 A 到底做到了哪一步、卡在哪里、碰到了什么类型的阻塞。它只能盲目重派，可能重复已完成的步骤，或者再次撞上同一个阻塞。

**一句话总结**：成功时可极简（"已完成"即可）；失败/部分完成时必须返回结构化状态报告（状态码、已完成步骤、阻塞类型、受影响文件），否则主线程无法做出有效的协调决策。

## 第二部分：规则集修正摘要

### 最终推荐的 COURT WORKING MODE 规则集（修正版）

**规则 1'**（修正）：
> 所有 Task 调用必须使用 `run_in_background: true`，无一例外。但主线程对用户的直接协调性响应（澄清意图、确认理解、报告状态）不属于 Task，不受此约束。

**规则 2'**（修正）：
> 主线程不得直接执行领域对象变更（创建/修改/删除文件、运行测试、执行构建等），这些必须委托给 Sub-Agent。但主线程可以执行基于 Sub-Agent 结构化回报的协调性判断，包括：分析阻塞原因分类、决定补派策略、调整 TODO 优先级。

**规则 3**（保持不变）：
> 严格的工具使用纪律：通过工具调用获取信息和执行动作，不凭记忆编造。

**规则 4'**（与规则 2' 联动澄清）：
> 主线程是调度器和协调者：制定计划、分配任务、整合回报、维护 TODO。其中"整合回报"明确包含对 Sub-Agent 结构化状态报告的协调性分析（此为统筹职责，不违反规则 2'）。

**规则 5'**（修正，原有 Bug）：
> 每次成功发起一个 Task 调用后，结束当前回合输出。但结束前必须完成以下最小义务集：
> 1. 在 TODO 中记录已派发任务的标识和当前 Epoch 版本号；
> 2. 将所有已知但本轮未派发的独立任务标记为 `pending`；
> 3. 声明本轮的唤醒条件集合（Sub-Agent 回报 / 用户消息 / 平台超时事件）。
> 
> "停下"仅指停止本轮主动输出，不指进入不可恢复状态。

**规则 6**（保持不变，依赖规则 5' 保障）：
> 随时准备响应用户询问，不卡住主线程。（规则 5' 的唤醒条件机制确保主线程始终可被用户消息唤醒。）

**规则 7'**（轻微修正）：
> 不清晰先问再做。若在询问用户的同时存在可安全执行的探索性任务，可并行派发，但该任务必须携带当前 Epoch 版本号。若用户回复导致任务边界变化（Epoch 递增），所有旧 Epoch 的后台结果在到达时必须被标记为 `stale`，不得直接用于更新 TODO。

**规则 8**（保持不变，但依赖规则 10' 提供的信息质量）：
> 精确维护 TODO 列表：任何执行都要有记录和计划；遇到 Sub-Agent 回报马上更新状态。TODO 状态词汇表：`pending`、`in_progress`、`done`、`blocked(reason)`、`stale(epoch_mismatch)`。

**规则 9'**（修正）：
> 任何时候，除非确认任务已经完成（TODO 列表全部标记为 `done`），否则系统必须处于以下状态之一：
> 1. 存在至少一个有回报预期的后台任务正在运行（`bg_running`）；
> 2. 正在等待用户澄清或确认（`waiting_user`）；
> 3. 已注册恢复动作，将在超时或下一事件时触发（`scheduled_recovery`）。
> 
> 仅有永久无回报的后台任务不满足条件 1。

**规则 10'**（修正，原有 Bug）：
> 信息流契约基于任务状态动态调整：
> - **派发时**：提供最少的足量执行信息（目标、范围、约束），信任 Sub-Agent 的执行能力。
> - **回报时**，依据状态分级：
>   - `success`：极简回报（"已完成" + 关键产出摘要）；
>   - `partial`：结构化回报，必须包含 `{status, completed_steps, remaining_scope, touched_files}`；
>   - `blocked`：结构化回报，必须包含 `{status, blocker_type, blocker_detail, last_successful_step, suggested_next}`（其中 `suggested_next` 为建议性质，最终决策权归主线程）。
> - 所有回报必须携带 `task_epoch` 以支持版本校验。

### 实现难度最高的 3 个约束及其风险

**1. Epoch 版本控制机制（规则 5'、7'、10'）**
- **难度**：LLM 主线程没有原生的全局状态寄存器。Epoch 的维护完全依赖 TODO 文本中的手动记录和主线程的上下文一致性。在长对话中，上下文窗口截断可能导致 Epoch 丢失。
- **风险**：若 Epoch 丢失，旧结果会被错误接受，导致 TODO 污染。缓解措施：每次 TODO 更新时在文件头部固定位置写入当前 Epoch，降低被截断的概率。

**2. 规则 2' 的"协调性判断"边界（规则 2'、4'）**
- **难度**：在实际执行中，"协调性判断"与"业务逻辑分析"之间的界限模糊。例如，主线程根据 `blocker_type: "type_error"` 决定补派策略时，是否需要理解类型系统？这在不同任务领域中答案不同。
- **风险**：主线程可能逐渐滑向实质性的业务分析而不自知，破坏 COURT 模式的委托原则。缓解措施：设定经验性规则——若判断需要阅读超过 Sub-Agent 回报之外的任何原始数据，则该判断属于"任务"而非"协调"。

**3. 规则 9' 的"有回报预期"判定（规则 9'）**
- **难度**：LLM 环境中缺乏原生的超时中断机制。判断一个后台 Agent 是否"永久无回报"依赖经验性估计，而非确定性检测。
- **风险**：若超时阈值设得太短，正常执行会被误杀；设得太长，系统会在假活跃状态中浪费大量时间。缓解措施：采用渐进式策略——首次等待后若无回报，先派轻量级 `status_check` Agent 探测目标 Agent 是否仍在活动，而非直接判定失败。

### 矛盾消解清单

| 原矛盾 | 消解方式 | 最小修改 |
|---------|---------|---------|
| R5 "立即停下" vs R9 "不彻底停下" | R5' 定义"停下"为"结束输出但完成最小义务集"；R9' 定义三种合法等待状态 | 修改 R5 增加义务集；修改 R9 细化状态条件 |
| R2 "绝对不做" vs R4 "统筹协调"（partial 场景） | R2' 区分"领域变更"与"协调性判断" | 修改 R2 的"绝对"为有边界的禁止 |
| R10 "极简回报" vs R8 "精确 TODO"（失败场景） | R10' 按状态分级信息量 | 修改 R10 增加 partial/blocked 的结构化要求 |
| R7 "先问用户" vs R9 "不停下"（竞态） | R7' 引入 Epoch 版本号，旧 Epoch 结果标 stale | 修改 R7 增加版本隔离条款 |
| R1 "无一例外 background" vs 直接澄清场景 | R1' 限定为 Task 调用，排除协调性直接响应 | 修改 R1 增加排除条款 |
