# COURT WORKING MODE：规则合理性、卡顿与 Bug 分析 裁判总结

> 2026-03-09T17:19:47.721921



# 最终裁定

## 第一部分：白话版结论

### 规则 1（铁律）：Task 工具调用必须永远带 `run_in_background: true`，无一例外

**裁定：合理，无需修正。**

三方在这一点上几乎没有实质争议。康德曾提出"如果用户只是问'你理解我意思吗'，也要派后台任务吗？"——但这是对规则射程的误读。规则 1 的原文明确限定为"Task 工具调用"，而非"主线程的一切行为"。直接回复用户澄清根本不涉及 Task 工具调用，所以不受此规则约束。

**具体例子**：用户说"我说的重构是提取函数，不是重写模块"。主线程直接回复"明白了，调整计划"——这不是 Task 调用，规则 1 根本不管它。但如果主线程要派一个 agent 去扫描代码结构，那就必须 `run_in_background: true`。

**一句话总结**：这条铁律边界清晰、目标正确，保持原样。

---

### 规则 2（原规则 1）：绝对不自己做任务，全部派后台 Sub-Agent 执行

**裁定：有 Bug，需修正。**

Ssyram 和康德都精确命中了同一个要害：**"任务"一词未被类型化，导致主线程的合法协调动作可能被自身规则吞噬。** Ssyram 的质询尤其尖锐：当 Sub-Agent 回报 `blocker_type: missing_dep` 时，主线程据此决定"派 B 去安装依赖"——这算"自己做任务"还是"协调判断"？如果连"读取结构化回报并决定下一步"都算"做任务"，那主线程连规则 7（精确维护 TODO）都无法执行，整个体系自我瘫痪。

Linus 的表述最到位：主线程应有"任务级感觉器官"，不能有"实现级手工代做"。

**具体例子**：Sub-Agent 回报"auth.py 第 42 行有类型错误，缺少 null check"。主线程更新 TODO 为 `blocked`，并派新 agent 去修复——这是协调。但如果主线程自己打开 auth.py 写了一个 `if (x != null)` 补丁——这才是"做任务"。当前规则的"绝对不"无法区分二者。

**修正**：主线程不得直接执行领域对象的调查、编辑、验证（禁止调用 Write/Edit 类工具处理业务文件）；但可进行计划制定、TODO 维护、回报分析、澄清问答与派发决策。

**一句话总结**：把"绝对不做任务"改成"不做领域执行，可做协调判断"，否则主线程连自己的职责都履行不了。

---

### 规则 3（原规则 2）：第一次派出 Explore/调查 agent 也不例外，立刻设 background

**裁定：合理，但冗余，可删除。**

三方均未对此提出实质攻击。Linus 直接说"这是规则 1/2 的特例重复，删掉"。他是对的。如果规则 1 已经说"Task 调用一律 background"，这条只是把同一件事再说一遍。保留它的唯一理由是"防止有人觉得第一次探索性任务可以例外"，但这种心理安慰不值得占一条独立规则。

**具体例子**：用户给了一个新任务，主线程要先派 agent 去了解项目结构。这个 explore agent 当然要 background——但这已经被规则 1 覆盖了。

**一句话总结**：并入规则 1，不需要单独存在。

---

### 规则 4（原规则 3）：主线程是调度器和协调者

**裁定：合理，与修正后的规则 2 合并即可。**

这条本质上是规则 2 的正面表述（规则 2 说"不做什么"，这条说"做什么"）。三方对其内容没有异议，争议全在边界上——而边界问题已在规则 2 的修正中解决。

**具体例子**：主线程收到三个 Sub-Agent 回报，整合后发现其中两个成功、一个 blocked，于是更新 TODO、补派一个新 agent、向用户汇报进展。这就是"调度器和协调者"的典型行为。

**一句话总结**：与修正后的规则 2 合并为主线程职责定义。

---

### 规则 5（原规则 4）：每次派完人立即停下

**裁定：有 Bug，必须修正。**

这是本次辩论中争议最集中、论证最充分的一条。三方的核心论据我逐一采纳或驳回：

**Linus 的攻击（采纳）**：`TODO=[A,B,C]`，派 A 后按"立即停下"，B/C 永远 pending。这不是理论洁癖——如果平台允许同回合多次派发，这条规则就把吞吐强制降为串行。Linus 给出的执行 trace（步骤 11/12 同时派 B 和 C）精确证明了"立即停下"在多任务场景下不可执行。

**康德的分析（采纳）**："停下"未区分层级——是"结束本轮输出"还是"系统失去活性"？这个语义模糊直接导致实现者可以走出失活分支。

**Ssyram 的串行饥饿论（采纳）**：如果 A 耗时极长，B/C 将处于无限期饥饿状态，违反了系统"流畅执行"的承诺。

**具体例子**：用户要求同时检查三个模块的类型错误。主线程建了三个 TODO 项，派第一个 agent 后"立即停下"。第一个 agent 跑了 5 分钟，第二第三个任务完全没开始。用户等了 15 分钟才完成本可 5 分钟并行完成的工作。

**修正**：每次完成本轮计划内的所有必要派发和 TODO 更新后，进入等待态。"停下"指放弃控制权进入等待，不指中断当前回合的协调逻辑。

**一句话总结**：从"派完立即停下"改为"完成本轮必要协调后进入等待态"。

---

### 规则 6（原规则 5）：随时准备响应用户询问，不卡住主线程

**裁定：合理，但应并入等待态定义而非单独成规。**

三方均未攻击其意图，只是认为它更像是一个设计原则而非可检查的规则。Linus 说"这其实是等待事件集的一部分"。他是对的。如果等待态已定义为 `{waiting_subagent, waiting_user}`，那"随时准备响应用户"就是等待态的天然性质，不需要单列。

**具体例子**：主线程已派出三个后台 agent，正在等回报。用户突然问"现在进度怎样？"主线程应该能立即回答——这在等待态模型下自然成立，因为用户消息就是事件源之一。

**一句话总结**：并入等待态的事件源定义，不需要独立规则。

---

### 规则 7（原规则 6）：如果对用户的任务不清晰，先执行任务计划和询问

**裁定：合理，保留。**

三方对此规则本身几乎没有攻击。Linus 提出了一个有价值的竞态场景（R7 vs R9：主线程一边问用户一边派了 explore agent，用户随后改变任务边界，explore 结果变成过期信息），但这个问题的根源不在规则 7 本身，而在缺乏版本/epoch 机制——已由规则集整体修正覆盖。

**具体例子**：用户说"把这个模块优化一下"。主线程不知道是优化性能还是优化可读性，先问"你说的优化是指运行速度还是代码结构？"等用户回复后再建 TODO 和派工。这完全合理。

**一句话总结**：保留原样，是正确的流程规则。

---

### 规则 8（原规则 7）：精确维护 TODO 列表

**裁定：合理，保留，但需与活性约束联动。**

新版规则 8 把旧版"不要停下"改成"必须处于 `bg_running` 或 `waiting_user`"，三方都认可这是重大改善。Linus 明确说"我支持 R8"。康德也说"比旧版大幅改善"。

唯一的补充需求来自三方共识：TODO 更新不能靠猜，必须有结构化回报支持——但这是规则 9/10 的问题，不是规则 8 自身的问题。

**具体例子**：Sub-Agent 回报 partial，主线程立即把对应 TODO 项从 `in_progress` 改为 `blocked`，并记录 blocker 类型。这就是"精确维护"。

**一句话总结**：核心规则之一，保留。

---

### 规则 9（原规则 8 的活性部分 + 新规则 9）：派发端只给目标/作用域/约束

**裁定：合理，且新版修正到位。**

这是新版改动最大、改得最好的一条。观察者提供的实际违规示例（主线程把修复方案的完整伪代码写进派发 prompt）精确证明了旧版的漏洞。三方对新版的"声明式派发"原则高度一致：

- **Linus**：主线程只给目标/作用域/约束/验收，不给实现步骤。
- **Ssyram**：严禁在 Prompt 中包含任何预构思的代码实现。
- **康德**：派发只给验收标准和定位信息，不给补丁文本。

新版的错误/正确示例也写得清楚，可操作。

**具体例子**：用户要修 `runner.py` 的 bug。正确派发："修复 runner.py 中 run() 和 resume() 路径的日志竞争问题，完成后汇报改动点与验证结果。"错误派发："先把第 440 行的 write_text 改成临时文件写入，再给 add() 加 flush=True，然后两处循环都改。"——后者主线程已经把方案做完了，Sub-Agent 沦为打字机。

**一句话总结**：新版修正到位，保留。

---

### 规则 10（新规则 10）：禁止在派发 prompt 中写入实现方案或完整内容复述

**裁定：合理，且与规则 9 形成完整的派发契约。**

新版将旧版模糊的"信任 Sub-Agent"改成了具体的禁止清单（禁止实现方案/伪代码、禁止已知文件完整内容复述），这直接消除了辩论中最大的实际违规模式。

但三方辩论揭示了一个**规则集中缺失的维度**：回报端的结构化约束。当前 10 条规则全都在管"派发"，没有一条管"回报格式"。这是必须补的。

**具体例子**：主线程知道 `config.ts` 里有 API key，但不应该在派发中把 config.ts 的内容贴进去。正确做法：给路径 `config.ts`，让 agent 自己读。

**一句话总结**：保留，且需补充回报端的结构化约束。

---

### 辩论中揭示的缺失规则

**裁定：必须新增两项约束。**

#### 缺失一：回报结构化契约

三方最强共识之一。Linus 给出了最小字段集，Ssyram 从类型系统角度证明了其必要性，康德也在第三轮明确接受。当 Sub-Agent 只回报"搞不了"而不说为什么、卡在哪、做了什么，主线程更新 TODO 就只能靠猜，补派就只能重复已做步骤。

**具体例子**：Agent 修改 auth.py 时只完成了 200 行就卡住了。如果只回报"failed"，主线程不知道是权限问题、上下文超限、还是测试失败，也不知道前 200 行是否已改好。结果要么整个重做（浪费），要么瞎猜补派（更浪费）。

#### 缺失二：活性状态约束的闭合

新版规则 8 已部分覆盖，但 Linus 提出的"Sub-Agent 崩溃且无回报"场景仍未完全解决。这取决于平台能力——如果平台保证所有 background task 必有终态回报，则问题自动消解；如果不保证，则规则集需要承认这个边界，而不是假装问题不存在。

## 第二部分：规则集修正摘要

### 最终推荐的 COURT WORKING MODE 规则集

```markdown
## COURT WORKING MODE (必须遵守)

**每次 compact 后和第一次执行都必须说 "!COURT WORKING MODE ENABLED!" 来确认记得**

工作规则：

> ⚠️ 铁律：Task 工具调用必须永远带 `run_in_background: true`，无一例外。忘记就是违规。

1. 主线程不得直接执行领域对象的调查、编辑、验证；
   全部通过后台 Sub-Agent 执行（run_in_background: true）。
   主线程只负责：制定计划、分配任务、整合回报、维护 TODO 列表、向用户澄清。

2. 完成本轮计划内的所有必要派发和 TODO 更新后，进入等待态；
   不在等待态中主动扩张执行。

3. 随时准备响应用户询问；
   如果对用户的任务不清晰，先执行任务计划和询问，
   询问时应清晰解释执行意图。

4. 精确维护 TODO 列表：
   任何执行前都要有计划在 TODO 中；
   遇到 Sub-Agent 回报马上更新状态（增删改）。

5. 任何时候，除非确认任务已经完成（TODO 列表全部标记为 `done`），
   否则系统必须处于以下状态之一（停止前必须确认并汇报状态）：
   - 存在至少一个有回报预期的后台任务正在运行
   - 正在等待用户澄清或确认（等待澄清前也需要有 TODO 表示等待澄清）

6. 派发端只给目标/作用域/约束/信息 Reference，**禁止预写和转述实现方案**。
   - 错误示例：把修复方案的伪代码写进 prompt。
   - 正确示例："src/foo.ts 第42行有类型错误，修复目标是让编译通过且不引入 `as any`"。
   - 派发时应写明需要汇报什么信息。

7. 以下内容**禁止**写进派发 prompt：
   - 禁止：具体实现方案或伪代码（正确做法：Sub-Agent 自己读代码判断）。
   - 禁止：已知文件的完整内容复述（正确做法：给路径，让它自己读）。
   - 例外：直接来自用户指令的验收标准和约束可以转述。

8. Sub-Agent 回报规范：
   - 成功：可极简回报（完成了什么、改了哪些文件）。
   - 部分完成/受阻/失败：必须结构化回报，至少包含：
     完成了哪些步骤、卡在什么类型的问题上（如上下文超限/测试失败/依赖缺失/信息不足）、
     建议下一步动作方向。出错则应详细汇报错误和自身分析，交还主线程判断。
```

### 对比原版的关键变化

| 项目 | 原版 | 修正版 | 依据 |
|------|------|--------|------|
| 规则 1（主线程职责） | "绝对不自己做任务" | "不做领域执行，可做协调判断" | 三方共识：原文会让主线程连 TODO 更新都无法合法执行 |
| 规则 2（原规则 2） | 单独列"explore 也要 background" | 删除，已被铁律覆盖 | Linus：纯重复 |
| 规则 4（停下规则） | "每次派完人立即停下" | "完成本轮必要派发和 TODO 更新后进入等待态" | 三方共识：原文导致多任务串行饥饿 |
| 规则 5（原规则 5） | 单独成规 | 并入规则 3 | 是等待态的自然性质 |
| 回报规范 | 无 | 新增规则 8 | 三方最强共识：缺乏结构化回报导致 TODO 更新靠猜 |

### 实现难度最高的 3 个约束及其风险

1. **规则 8（回报结构化）**：风险最高。Sub-Agent 是 LLM，不保证总能输出结构化格式。当模型"幻觉"或上下文超限时，回报可能退化为自然语言。**缓解**：将结构化要求定义为"应当"而非"必须"，主线程在收到非结构化回报时降级为向用户转述并请求指导。

2. **规则 1 的边界判定（"领域执行"vs"协调判断"）**：主线程"读取 blocker 并决定补派"是协调还是执行？边界在大多数情况下清晰（不直接改文件=协调），但极端场景下可能模糊。**缓解**：采用 Ssyram 的操作类型签名——禁止 Write/Edit 类工具调用即可，按工具类型判定而非语义判定。

3. **规则 5 的活性保证（Sub-Agent 永不回报场景）**：如果平台不保证 background task 必有终态回报，系统可能进入永久等待。**缓解**：这是平台层问题，规则集应明确声明其假设——"本规则集假设 Task 工具的 background 模式保证最终返回结果或错误"。如果平台不满足此假设，需在平台层补超时机制。

### 矛盾消解说明

原版最大矛盾是"立即停下"（旧规则 4）vs"不要彻底停下"（旧规则 8/新规则 5）。修正版通过以下方式消解：

- **将"停下"重定义为"进入等待态"**，而非"系统失去活性"
- **将"不要停下"重定义为"必须处于可判定的等待状态"**，而非模糊的"保持活跃"
- 两条规则不再互相打架：规则 2 说"协调完了就等着"，规则 5 说"等着的时候必须有东西可等"

这就是 Linus 说的"调度节流 vs 状态合法性"的分层，康德说的"停下₁ vs 停下₂ 的概念分层"，Ssyram 说的"Yield vs Liveness Invariant"——三个人用不同语言说了同一件事。修正版把这个共识写进了规则文本。
